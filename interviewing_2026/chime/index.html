    Pattern 1: Hashmap for "have I seen this?"

    Recognize: "Find pair/group with property X" — anything checkable in O(1).

    def two_sum(nums, target)
  seen = {}
  nums.each_with_index do |num, i|
    complement = target - num
    return [seen[complement], i] if seen.key?(complement)
    seen[num] = i
  end
  []
end
    • Trap: Store after checking complement, not before (avoids matching self).
    • Complexity: O(n) time, O(n) space.

    Pattern 2: Prefix sum + hashmap

    Recognize: "Contiguous subarray sum equals/divisible by k" — especially with negatives.

    def subarray_sum(nums, k)
  count = 0
  prefix_sum = 0
  sums = Hash.new(0)
  sums[0] = 1  # phantom prefix — empty subarray sums to 0
  nums.each do |num|
    prefix_sum += num
    count += sums[prefix_sum - k]
    sums[prefix_sum] += 1
  end
  count
end
    • Trap: Forgetting sums[0] = 1 — misses subarrays starting at index 0.
    • When NOT to use: All-positive arrays → sliding window is simpler.

    Bank account analogy (Chime-friendly): Running balance = prefix sum. Hashmap = notebook of past balances. "Has my balance ever been (today's balance − k)?" = "is there a stretch where I netted exactly k?" Drop this — it's on-domain.

    Pattern 3: Sliding Window

    Three flavors

    Shrink whenInvariant brokenInvariant holds

    O(n) amortized: both pointers move ≤ n times total. Verbalize this.

    Longest Substring Without Repeating

    def length_of_longest_substring(s)
  last_seen = {}
  left = 0
  max_len = 0
  s.each_char.with_index do |char, right|
    if last_seen.key?(char) && last_seen[char] >= left
      left = last_seen[char] + 1
    end
    last_seen[char] = right
    max_len = [max_len, right - left + 1].max
  end
  max_len
end

    Trap: >= left check (skip already-evicted duplicates).

    Longest Substring with At Most K Distinct

    def longest_substring_k_distinct(s, k)
  return 0 if k == 0
  counts = Hash.new(0)
  left = 0
  best = 0
  s.each_char.with_index do |char, right|
    counts[char] += 1
    while counts.size > k
      counts[s[left]] -= 1
      counts.delete(s[left]) if counts[s[left]] == 0
      left += 1
    end
    best = [best, right - left + 1].max
  end
  best
end

    Trap: Delete zero-count keys, or counts.size overcounts distinct chars.

    Minimum Window Substring (variable shortest)

    def min_window(s, t)
  return "" if t.length > s.length
  required = Hash.new(0)
  t.each_char { |c| required[c] += 1 }
  needed = required.size
  have = Hash.new(0)
  formed = 0
  left = 0
  best_len = Float::INFINITY
  best_left = 0
  s.each_char.with_index do |char, right|
    have[char] += 1
    formed += 1 if required.key?(char) && have[char] == required[char]
    while formed == needed
      if right - left + 1 < best_len
        best_len = right - left + 1
        best_left = left
      end
      left_char = s[left]
      have[left_char] -= 1
      formed -= 1 if required.key?(left_char) && have[left_char] < required[left_char]
      left += 1
    end
  end
  best_len == Float::INFINITY ? "" : s[best_left, best_len]
end

    Key trick: formed counter updates in O(1). Don't compare hashmaps each step.

    Pattern 4: Sort + two pointers

    Recognize: Pairs/triplets/quadruplets summing to a target.

    def three_sum(nums)
  nums.sort!
  result = []
  n = nums.length
  (0...n - 2).each do |i|
    next if i > 0 && nums[i] == nums[i - 1]  # dedup anchor
    left, right = i + 1, n - 1
    while left < right
      total = nums[i] + nums[left] + nums[right]
      if total < 0
        left += 1
      elsif total > 0
        right -= 1
      else
        result << [nums[i], nums[left], nums[right]]
        left += 1
        right -= 1
        left += 1 while left < right && nums[left] == nums[left - 1]
        right -= 1 while left < right && nums[right] == nums[right + 1]
      end
    end
  end
  result
end

    Trap: Three dedup spots — anchor, left after match, right after match. Complexity: O(n²) time, O(1) extra space.

    Pattern 5: Tree DFS with bounds

    Recognize: Validity depends on ancestral context, not just parent-child.

    def is_valid_bst(root, low = -Float::INFINITY, high = Float::INFINITY)
  return true if root.nil?
  return false if root.val <= low || root.val >= high
  is_valid_bst(root.left, low, root.val) &&
    is_valid_bst(root.right, root.val, high)
end

    Trap: Local parent-child checks miss cases like a 4 nested deep in 5's right subtree. O(n) time, O(h) stack.

    Pattern 6: Tree BFS by level

    Recognize: "Process level by level / right side view / zigzag / shortest path on tree."

    def level_order(root)
  return [] if root.nil?
  result = []
  queue = [root]
  until queue.empty?
    level = []
    queue.size.times do  # snapshot size BEFORE inner loop
      node = queue.shift
      level << node.val
      queue << node.left if node.left
      queue << node.right if node.right
    end
    result << level
  end
  result
end

    Trap: Reading queue.size during the inner loop — snapshot it first. Ruby gotcha: Array#shift is O(n); fine for interview-sized inputs.

    Pattern 7: Grid traversal (DFS/BFS)

    Recognize: Connected components on a 2D grid.

    def num_islands(grid)
  return 0 if grid.empty?
  rows, cols = grid.length, grid[0].length
  count = 0
  rows.times do |r|
    cols.times do |c|
      if grid[r][c] == '1'
        count += 1
        dfs(grid, r, c, rows, cols)
      end
    end
  end
  count
end

def dfs(grid, r, c, rows, cols)
  return if r < 0 || r >= rows || c < 0 || c >= cols || grid[r][c] != '1'
  grid[r][c] = '0'  # mark visited by mutating
  dfs(grid, r + 1, c, rows, cols)
  dfs(grid, r - 1, c, rows, cols)
  dfs(grid, r, c + 1, rows, cols)
  dfs(grid, r, c - 1, rows, cols)
end

    Senior move: Ask "can I mutate the grid?" before starting. If no, use a visited Set.

    Pattern 8: Stack matching

    Recognize: Nested or paired structure.

    def is_valid(s)
  pairs = { ')' => '(', ']' => '[', '}' => '{' }
  stack = []
  s.each_char do |c|
    if pairs.key?(c)
      return false if stack.empty? || stack.pop != pairs[c]
    else
      stack.push(c)
    end
  end
  stack.empty?
end

    Trap: Three failure modes — empty stack on close, mismatch on close, leftover opens at end.

    Pattern 9: Heap / bucket sort for top-K

    Recognize: "Top K frequent / largest / smallest." Ruby has no built-in heap.

    # O(n log n) — sort, simplest
def top_k_frequent(nums, k)
  counts = Hash.new(0)
  nums.each { |n| counts[n] += 1 }
  counts.sort_by { |_, c| -c }.first(k).map(&:first)
end

# O(n) — bucket sort, mention as the elegant alternative
def top_k_frequent_bucket(nums, k)
  counts = Hash.new(0)
  nums.each { |n| counts[n] += 1 }
  buckets = Array.new(nums.length + 1) { [] }
  counts.each { |num, c| buckets[c] << num }
  result = []
  buckets.reverse_each do |bucket|
    bucket.each do |num|
      result << num
      return result if result.length == k
    end
  end
  result
end

    Bucket sort works because frequency is bounded by nums.length — index-as-frequency.

    OOD Problem 1: LRU Cache

    Pattern: Hashmap (O(1) lookup) + doubly linked list (O(1) reorder).

    Ruby-clever (mention as senior aside)

    class LRUCache
  def initialize(capacity)
    @cap = capacity
    @cache = {}
  end

  def get(key)
    return -1 unless @cache.key?(key)
    val = @cache.delete(key)
    @cache[key] = val  # re-insert moves to end (most recent)
    val
  end

  def put(key, value)
    @cache.delete(key) if @cache.key?(key)
    @cache[key] = value
    @cache.shift if @cache.size > @cap  # removes oldest
  end
end

    Classical (what they want to see)

    class LRUCache
  class Node
    attr_accessor :key, :val, :prev, :next
    def initialize(key = 0, val = 0)
      @key = key; @val = val; @prev = nil; @next = nil
    end
  end

  def initialize(capacity)
    @cap = capacity
    @cache = {}
    @head = Node.new  # sentinel: most recent side
    @tail = Node.new  # sentinel: least recent side
    @head.next = @tail
    @tail.prev = @head
  end

  def get(key)
    return -1 unless @cache.key?(key)
    node = @cache[key]
    remove(node)
    add_to_front(node)
    node.val
  end

  def put(key, value)
    remove(@cache[key]) if @cache.key?(key)
    node = Node.new(key, value)
    @cache[key] = node
    add_to_front(node)
    if @cache.size > @cap
      lru = @tail.prev
      remove(lru)
      @cache.delete(lru.key)  # critical: also remove from hashmap
    end
  end

  private

  def remove(node)
    node.prev.next = node.next
    node.next.prev = node.prev
  end

  def add_to_front(node)
    node.next = @head.next
    node.prev = @head
    @head.next.prev = node
    @head.next = node
  end
end

    TALKING POINTS

    1. Doubly linked, not singly — need O(1) removal of arbitrary node.
    2. Sentinel head/tail eliminate null checks at boundaries.
    3. Store key on node so put can also delete from hashmap on evict.
    4. Mention the Ruby Hash trick as a senior aside.

    Follow-ups: Thread safety → Mutex#synchronize. Distributed → Redis with EXPIRE.

    OOD Problem 2: Logger Rate Limiter

    Approach 1: Simple, unbounded memory

    class Logger
  WINDOW = 10

  def initialize
    @last_seen = {}
  end

  def should_print_message(timestamp, message)
    if !@last_seen.key?(message) || timestamp >= @last_seen[message] + WINDOW
      @last_seen[message] = timestamp
      true
    else
      false
    end
  end
end

    Approach 2: Bounded memory with sliding window

    require 'set'

class Logger
  WINDOW = 10

  def initialize
    @queue = []         # [timestamp, message] pairs
    @recent = Set.new   # O(1) membership lookup
  end

  def should_print_message(timestamp, message)
    while !@queue.empty? && @queue.first[0] <= timestamp - WINDOW
      _, old_msg = @queue.shift
      @recent.delete(old_msg)
    end
    return false if @recent.include?(message)
    @queue << [timestamp, message]
    @recent.add(message)
    true
  end
end

    Fintech-specific essentials (always mention these for Chime)

    • Idempotency. Every state-changing API takes an Idempotency-Key header. Dedupe at the API or app layer with a (key, request_hash) table.
    • Double-entry ledger. Money moves are debits + credits in the same transaction. Sum of every account = 0. Audit trail by construction.
    • Exactly-once via transactional outbox. Don't dual-write to DB + queue. Write a row to events in the same DB txn; a poller publishes it. (Or use Debezium / CDC.)
    • Compensating transactions. No 2PC across services. If step 2 fails, run a compensation for step 1.
    • Audit log immutable + append-only. Regulators (and your fraud team) need to reconstruct history.
    • Encryption. PII / PAN at rest (KMS-backed envelope) and in transit (TLS). Tokenize card numbers.
    • PCI scope minimization. Card data never lands in your own DB if you can route through a tokenization vault.
    • Reconciliation jobs. Daily diff against partner bank statements. Alert on drift.
    • Rate limiting on sensitive endpoints. Login, password reset, transfer-initiate.

    Requirements (clarify first)

    • Functional: Member A sends $X to member B. Both see balance update. Notifications fire. Idempotent retries.
    • Non-functional: p99 < 500ms perceived. Strong consistency on balances (no double-spend). 99.99% availability for the API. Audit trail forever.

    Capacity (anchor with numbers)

    ~25M members. Assume 5% DAU send/day = 1.25M transfers/day = ~15 TPS average, ~150 TPS peak. Cheap; we're correctness-bound not throughput-bound.

    API

    POST /v1/transfers
  Headers: Idempotency-Key: <uuid>, Authorization: Bearer ...
  Body: { from_member_id, to_member_id, amount_cents, currency, memo }
  Response: { transfer_id, status, ledger_entries }

    Data model

    accountsid, member_id, balance_cents (cache only — derive from ledger), updated_at
    ledger_entriesid, account_id, transfer_id, amount_cents (signed), created_at — append-only
    transfersid, idempotency_key (UNIQUE), from, to, amount, status, created_at
    idempotency_keyskey, request_hash, response_blob, created_at — TTL 24h

    Write path (the meat)

    1. API receives request. Look up idempotency_keys: if hit and request matches → replay stored response. If hit but request differs → 422 conflict.
    2. Open DB transaction. Insert transfers row (UNIQUE on idempotency_key catches racing requests).
    3. Insert two ledger_entries: (from, -amount), (to, +amount).
    4. Update accounts.balance_cents for both (with a CHECK constraint: balance >= 0 — if it fails, the txn rolls back and we return 402).
    5. Insert events row for outbox (notification fanout).
    6. Commit. Store response in idempotency_keys. Return 200.

    Async path

    Outbox poller reads events → publishes to SNS/Kafka → notification service → push + email + in-app.

    Failure modes (volunteer these)

    • Network blip on response: client retries with same Idempotency-Key → replay path.
    • DB primary failover mid-transaction: txn rolls back; client sees 5xx and retries; idempotency key dedupes.
    • Concurrent transfers from same account: row-level lock on the account (SELECT ... FOR UPDATE) or rely on the CHECK constraint to fail one of them.
    • Outbox publisher crashes after commit but before publish: next poll sends — at-least-once delivery downstream, consumer must be idempotent.
    • Notification service down: doesn't block the transfer. Retry with backoff; surface in in-app feed eventually.

    Scaling notes

    At 150 TPS, single Postgres primary is fine for years. If we needed to scale: shard by from_member_id, accept that cross-shard transfers need a saga. Read replicas for balance reads (with read-your-writes via session pinning).

    Likely problem: Idempotency layer (cross-service)

    Stripe-style framing. They love this because it's directly applicable to their domain.

    Storage

    Redis (fast, TTL) for hot keys + Postgres (durable) for audit. Or single Postgres with table partitioned by created_at day.

    Schema

    idempotency_keys (
  key VARCHAR PRIMARY KEY,
  request_hash VARCHAR NOT NULL,
  response_status INT,
  response_body JSONB,
  state ENUM('in_flight', 'completed'),
  locked_until TIMESTAMP,
  created_at TIMESTAMP
)

    Flow

    1. Insert row with state='in_flight', locked_until=NOW()+30s. ON CONFLICT (key):
    2. If existing is completed + hashes match → return cached response.
    3. If existing is completed + hashes differ → 422.
    4. If existing is in_flight + lock not expired → 409 (retry later).
    5. If existing is in_flight + lock expired → take it over (assume prior request died).
    6. Process request, update row to completed with response.

    Senior touches: hash the canonicalized request body to detect divergence. TTL keys after 24h. Don't accept duplicate keys with different request payloads (that's a client bug).

    Likely problem: Rate limiter (sensitive endpoints)

    Requirements

    Throttle login attempts, password resets, transfer-initiate by (member, IP, device). Fail closed.

    Algorithm: token bucket in Redis

    key = "rl:#{bucket_id}"
# Lua script for atomicity:
#   tokens = GET key (or capacity if missing)
#   refill based on time elapsed
#   if tokens >= 1: decrement, allow
#   else: deny + return retry-after

    Tradeoffs

    • Sliding-window log = precise but memory O(requests per window). Use for low-traffic / very sensitive endpoints.
    • Fixed window = simple, burst-prone at boundaries. Use for analytics endpoints, not money.
    • Token bucket = smooth, bursty allowance. Default choice.
    • Distributed: Redis cluster, single key per bucket → potential hot spot. Mitigate with key sharding by IP-suffix or load-shedding before Redis.

    Failure mode

    Redis down → fail closed for money APIs (better to reject than allow unbounded). Fail open only for non-sensitive reads.

    Likely problem: Transaction ledger (ChimeCore-flavored)

    This may come up explicitly. ChimeCore is their proprietary ledger and transaction-processing platform. Show you understand double-entry.

    Core invariant

    Every transaction creates 2+ ledger entries summing to zero. SUM(ledger_entries.amount_cents) GROUPED BY transaction_id = 0. Always.

    Schema

    accounts (id, owner_id, type, currency)
  type: member_checking, member_savings, partner_bank_settlement,
        fee_revenue, interchange_revenue, ...

ledger_entries (
  id, account_id, transaction_id, amount_cents (signed),
  posted_at, created_at
) -- append-only, never UPDATE

transactions (
  id, kind, status, idempotency_key, external_ref, created_at
)

    Operations

    • Post: Insert N ledger entries in one DB txn. Reject if sum != 0.
    • Balance: SUM(amount_cents) WHERE account_id = ?. Materialize a cached balance on a separate row, updated in same txn — but the ledger is the source of truth.
    • Reverse: New transaction with negated entries. Never delete historical entries.
    • Reconcile: Daily job compares SUM(member_checking) against partner bank's reported balance. Alert on drift.

    Scaling

    Partition ledger_entries by posted_at month → fast pruning of cold partitions to cheaper storage (S3 / cold Postgres). Materialized balance view refreshed in-line with each post.

    Story 6: Disagreement / pushback (DDD on Zendesk Talk usage payments)

    Pushed back on DDD mandate for a small project — deferred, was honestly both right and wrong
    chime insimplifytrust
    Situation
    At Zendesk I owned the Talk usage payments project — relatively small scope, small team, me as sole engineering owner on the implementation. Around that time the architecture group was standardizing on Domain-Driven Design as the pattern for payments work across the broader org.
    Tension
    I thought DDD was overkill for my project — the scope didn't justify the ceremony of aggregates, ubiquitous language, bounded-context boundaries. My second concern was forward-looking: a future engineer parachuting in to extend a small service wrapped in DDD scaffolding would have a harder onboarding than necessary. The architecture group's standing wasn't nothing — they were optimizing for org-wide consistency and shared vocabulary across the broader payments surface, which is a legitimate goal I wasn't carrying.
    Action
    I raised it in 1:1s with the arch lead, not in the architecture review meeting — better signal-to-noise, less ego. I made the case on scope and team-size grounds and offered a lighter-weight alternative. I genuinely tried their position rather than just objecting — implemented enough of it to feel the friction firsthand. Once it was clear the org-consistency argument was going to win, I deferred and committed. To mitigate my discoverability concern, I went heavy on inline comments and wrote thorough READMEs so the next engineer didn't have to reverse-engineer the DDD mapping from the code alone.
    Result
    Honestly, both right and wrong. Right: the DDD ceremony did add cost that wouldn't have existed at this scope, and I've seen engineers since then take longer to extend it than they would have a simpler version. Wrong: the org-wide consistency the arch group cared about actually paid off — when other payments work touched my service, the shared vocabulary made cross-team conversations dramatically easier than they'd have been if I'd done my own thing. The READMEs and comments turned out to be the single highest-leverage thing I shipped on that project; they outlived multiple engineer transitions.
    Reflection
    Two things I carry from it. First — when I argue scope vs. shared standards now, I steelman the network effects of consistency much harder before I push back. I undervalued it then. Second — when I lose a call I disagreed with, the work isn't to be quietly bitter; it's to find the mitigation that makes the chosen path work as well as it can. The doc-heavy mitigation was the part I'm most proud of in retrospect, not the pushback itself.
    Chime angle
    "'Chime in' is owed when you disagree — but so is recommitting hard when the call lands somewhere you didn't choose. The mitigation work after a deferred disagreement is where the real character shows up. And the README/comments lesson is exactly the kind of thing I'd bring to ChimeCore-adjacent work — make the next engineer's onboarding the design constraint, not the afterthought."

    Story 7: Ambiguity (order processing service)

    Built order processing integration service from a one-line spec

    Situation
    At Zendesk the ask started as one line — "we need a service that handles orders end-to-end." No spec, no design doc.

    The thing I'd do differently: structured metrics on day one. I had logs but not dashboards, and the gap cost me hours the first time something went sideways in prod.
