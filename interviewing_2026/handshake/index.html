    Before each round (30-second prep)

    LEADERSHIP

    • Have 3 stories ready: leadership crisis, cross-functional, failure + learning
    • Tie each to one Handshake value
    • End each with "and here's what I'd do differently"

    PAIRING

    • Ask 2 clarifiers in the first 90 seconds
    • Verbalize the data model before any code
    • Run / test after every function — never go silent > 30s
    • Treat the interviewer as a teammate, not a judge

    Red flags to avoid (any round)

    • "I" instead of "we" for collaborative work — and vice versa for things you owned
    • Blaming managers, teams, or "the deadline" — always own the part you could have changed
    • Generic "great company" answers — they want to hear Handshake-specific knowledge
    • Silent coding in pairing — > 30 seconds of dead air is a real ding
    • Defensive responses to feedback — "good point, let me try that" beats "actually…"
    • Skipping the design discussion in pairing — talk before typing
    • Pure-tech humblebrag in leadership — "I rewrote the whole system alone" = anti-signal

    When stuck (any round)

    • Restate the problem aloud. 70% of bugs are misread requirements.
    • Trace one example by hand. "Let me walk through input X step by step."
    • Ask the interviewer. "I want to verify my assumption — can you confirm Y?"
    • Out loud: "Let me try a simpler version first." Then iterate.
    • Behavioral version: "Let me think about that for a moment" — silence is fine for 5–10 seconds in the leadership round, just not in the coding round.

    Leadership Round conversational, project-focused, value-aligned

    Format & what to expect

    • Format: Usually 1:1 with hiring manager (sometimes +1 senior IC or PM). Conversational, not a panel grill.
    • Length: 45–60 min
    • Style: Project-focused. Candidates report this is "less standard behavioral, more centered around your interests and the projects you've worked on."
    • What they're really checking: Will you fit "Olympic pace"? Do you genuinely care about the mission? Can you self-direct in ambiguity?

    First 5 minutes (do this every time)

    1. Read the prompt twice. Don't start typing.
    2. Restate it back. "So I'm building X that supports Y, input format is Z. Did I capture that right?"
    3. Ask 1–2 high-value clarifiers. Examples:
      • "Is input a stream of method calls or string commands I parse?"
      • "On error, do you want me to raise, return None, or log and continue?"

    4. State the data model in one sentence. "I'll model this as a class with a dict of jobs keyed by id."
    5. Type the empty skeleton. Class + one method stub + one inline assert. Run it. Now implement.

    The problem playbook (works on 90% of CodeSignal projects)

    1. Define the domain object. Struct.new(:a, :b, keyword_init: true) — gives you accessors, ==, and to_s for free.
    2. Define a container. Almost always Database / Manager / Registry class with a dict inside.
    3. Stub the public methods. Empty bodies, type hints, one assert per method that runs.
    4. Implement the simplest method first. Usually create / insert / register.
    5. Run after every method. Inline assertions, not a test framework, unless they offer one.
    6. Handle errors specifically. Raise typed exceptions with messages. raise ValueError(f"no such job: {job_id}").
    7. When Part 2 lands: acknowledge aloud, decide refactor-vs-extend, then do it.
    8. End with edge cases volunteered. "Empty input, duplicate IDs, missing fields — let me handle those quickly."

    Language: Ruby

    You're coding in Ruby. Best possible signal for a Rails shop — they'll feel at home reading your code, and you'll feel at home writing it.

    Defaults to lean on:

    • Struct.new(:a, :b, keyword_init: true) for value objects

    Iteration when they add a part

    1. Acknowledge the new requirement aloud before touching code. "Got it — so now we need to support partial applications. Behavior is X."
    2. Ask: does this change existing behavior, or only add to it? Determines refactor vs. extend.
    3. If refactor needed: say it. "To do this cleanly I want to pull this if-chain into a strategy. Two minutes."
    4. Re-run all prior asserts after each part. Out loud: "Let me re-run Part 1's checks to make sure I didn't regress."

    Tier 1 — Vending Machine OOP

    Verbatim prompt (Glassdoor + 1Point3Acres):

    "Code a vending machine that needs to track inventory and return correct change in denominations of 5, 10, and 25 cents."

    Reported follow-on requirement:

    "Two methods are needed — one to input change, one to select an item — and the two actions should be idempotent and able to happen in any order before vending."

    What makes it hard: the idempotency + ordering twist. Most candidates write a stateful "insert coin → pick item → vend" linear flow and get caught by Part 2 when they're told the operations can happen in any order and may be repeated safely.

    Design checklist before you code

    • State: inventory (dict: item → (price_cents, count)), coin balance currently inserted (int), accumulated revenue, change drawer (dict: denomination → count)
    • Public API: insert_coin(cents), select_item(name), refund(), restock(name, qty, price_cents)
    • Idempotency model (Part 2): if "select" is called twice for the same item before vending, treat the second call as a no-op (or a re-select with the same intent). If insert is called multiple times before select, accumulate.
    • Change-making: greedy with 25→10→5 works here (it's a canonical change set). If they swap denominations later (e.g., add 7-cent coins), be ready to switch to DP and call that out.
    • Errors: out-of-stock → return inserted coins; insufficient funds → keep coins, tell user how much more; can't make exact change → refund or refuse.

    Vending Machine — starter code (Ruby)

    Each public method does one thing. vend reads top-to-bottom like the spec. Trace one example through it to lock in the flow.

    Mental model — read this first

    • 4 pieces of state: @items (what's for sale), @drawer (coins on hand for change), @paid (this round's coins), @choice (what the user picked).
    • 2 user actions: insert(coin) bumps @paid. select(name) sets @choice. Both are idempotent and order-independent — that's the Part 2 twist.
    • 1 commit action: vend runs all the checks; if everything passes, it decrements stock + drawer and returns [name, change].
    # Greedy change-making works because each denom divides the next (5 | 10 | 25).
# For arbitrary denoms, switch to DP — mention this aloud, don't code unless asked.
DENOMS = [25, 10, 5].freeze

Item = Struct.new(:name, :price, :stock, keyword_init: true)

class VendingMachine
  def initialize
    @items  = {}             # name => Item
    @drawer = Hash.new(0)    # denom => count of coins on hand
    @paid   = 0              # cents inserted this round
    @choice = nil            # item name the user picked (nil = none yet)
  end

  # ---------- setup ----------

  def restock(name:, price:, stock:)
    @items[name] = Item.new(name: name, price: price, stock: stock)
  end

  def load_drawer(denom, count)
    @drawer[denom] += count
  end

  # ---------- user actions (idempotent + any-order) ----------

  def insert(coin)
    raise ArgumentError, "bad coin: #{coin}" unless DENOMS.include?(coin)
    @drawer[coin] += 1
    @paid += coin
  end

  def select(name)
    raise KeyError, "no item: #{name}" unless @items.key?(name)
    @choice = name                       # re-selecting is a no-op
  end

  def refund
    refunded = @paid
    reset_round
    refunded
  end

  # ---------- vend ----------
  # Reads like the spec: get the item, check funds, compute change, commit.
  def vend
    item   = chosen_item                 # raises if no choice / out of stock
    short  = item.price - @paid
    raise "need #{short} more cents" if short.positive?

    change = make_change(@paid - item.price)
    raise "can't make exact change" if change.nil?

    commit_purchase(item, change)
  end

  private

  def chosen_item
    raise "no item selected" if @choice.nil?
    item = @items[@choice]
    raise "out of stock: #{item.name}" if item.stock.zero?
    item
  end

  # Validate-then-commit: only mutate state once we know the sale will succeed.
  def commit_purchase(item, change)
    change.each { |d, n| @drawer[d] -= n }
    item.stock -= 1
    bought = item.name
    reset_round
    [bought, change]
  end

  def reset_round
    @paid   = 0
    @choice = nil
  end

  def make_change(cents)
    out = {}
    DENOMS.each do |d|                   # walk denoms largest → smallest
      n = [cents / d, @drawer[d]].min    # use what's needed OR what's available
      next if n.zero?
      out[d] = n
      cents -= d * n
    end
    cents.zero? ? out : nil              # nil = couldn't cover the amount
  end
end

    What vend is doing, in plain English

    1. Get the chosen item. Raise if nothing's been selected or it's out of stock.
    2. Check funds. If short, raise with how much more is needed.
    3. Compute change. If the drawer can't cover it, raise — without mutating anything.
    4. Commit. Subtract change from drawer, decrement stock, reset round, return [name, change].

    The reliability trick: all the checks happen before any state changes. If make_change returns nil, we raise and nothing has been mutated. No partial sale, no inventory in a weird state. This is the "validate-then-commit" pattern — say it aloud when you write commit_purchase; interviewers love hearing it.

require "time"   # stdlib Time helpers; not strictly needed for the inline tests

# Struct.new + block lets us attach instance methods to a Struct.
# Why: Post stays mostly value-y, but `post.upvote!` reads better
# than `post.upvotes += 1` scattered throughout the Feed class.
# The bang (!) is Ruby convention for "mutates self".
Post = Struct.new(:id, :title, :author, :created_at, :upvotes, keyword_init: true) do
  def upvote!
    self.upvotes += 1
  end
end

class Feed
  # Constants pulled out for two reasons:
  #   1. self-documenting — `window: ONE_DAY` reads better than `window: 86400`
  #   2. trivial to swap if interviewer says "make it 1 hour"
  # Ruby's Time arithmetic uses SECONDS, so all time math is in seconds.
  ONE_DAY = 24 * 60 * 60

  def initialize
    @posts = {}   # id => Post — same Hash-keyed-by-id workhorse pattern as before
  end

  # ---------------- CRUD ----------------

  def create(post)
    # Defensive: refuse duplicate ids. Caller probably has a bug.
    raise ArgumentError, "duplicate post: #{post.id}" if @posts.key?(post.id)
    # `||=`  assigns only if current value is nil/false. Safer than `= 0`
    # because it preserves an explicitly-passed value (e.g. 0 stays 0,
    # 42 stays 42, but nil becomes 0).
    post.upvotes ||= 0
    @posts[post.id] = post
  end

  # Hash#fetch with a BLOCK: the block runs only if the key is missing.
  # Why prefer this over `@posts[id] or raise ...`?
  #   - `nil` could be a legitimate value in other hashes
  #   - the block-form is the idiomatic "fail loudly with a message" pattern
  def get(id)
    @posts.fetch(id) { raise KeyError, "no such post: #{id}" }
  end

  # Delegate to get() so the "no such post" message is DRY across callers.
  def upvote(id)
    get(id).upvote!
  end

  # Hash#delete returns the deleted value, or nil if missing.
  # We use that nil as the trigger for the raise — concise.
  def delete(id)
    @posts.delete(id) || (raise KeyError, "no such post: #{id}")
  end

  # ---------------- PART 2: ranked feed ----------------
  #
  # The brief "most recent + most upvoted at the top" is AMBIGUOUS.
  # Real-world it could mean any of:
  #   (a) Sort by upvotes within a recency window         ← we implement this
  #   (b) Compute a hot-score combining age + upvotes     ← see self.hot_score
  #   (c) Strict "most recent first, tiebreak by upvotes" ← different sort key
  # ALWAYS ASK which one before coding. The asking is half the score.
  #
  # `now:` is a KEYWORD ARG (not hardcoded Time.now). Why:
  #   tests can pass a fixed `now` so the result is deterministic.
  #   Mention this aloud — interviewers notice testability awareness.
  #
  # `window:` and `limit:` default-arg'd so callers can override.
  def feed(now: Time.now, window: ONE_DAY, limit: 20)
    # Filter to recent posts.
    # Time subtraction returns a Float of seconds elapsed.
    # `select` returns a NEW array — non-mutating, doesn't touch @posts.
    recent = @posts.values.select { |p| (now - p.created_at) <= window }

    # ============= THE KEY IDIOM: tuple-key sort_by =============
    #
    # sort_by returns ascending order. To sort DESC on a numeric field,
    # negate it: `-p.upvotes` puts the highest-upvoted post FIRST.
    #
    # The tuple [a, b, c] compares lexicographically — first by a,
    # then by b only on ties, then by c only on tie-of-ties. So this:
    #
    #     [-p.upvotes, -p.created_at.to_i, p.id]
    #
    # expresses, in plain English:
    #   "primary: upvotes desc; tiebreak: recency desc; final tiebreak: id asc"
    #
    # Why include `p.id` as the LAST tiebreak? DETERMINISM.
    # If two posts have identical upvotes AND identical timestamps,
    # we still want the same ordering across runs (for tests, for caching,
    # for pagination cursors). Mentioning this signals seniority.
    #
    # `.first(limit)` after sort: take the top N. (`.take(limit)` would
    # also work and is identical here. Both are idiomatic.)
    recent.sort_by { |p| [-p.upvotes, -p.created_at.to_i, p.id] }.first(limit)
  end

  # ---------------- Part 3 alt: hot-score (Reddit/HN-style) ----------------
  #
  # DON'T code this unprompted. But MENTION the concept out loud:
  #   "Another interpretation of 'recent + popular' is a single hot-score
  #    that combines age + upvotes — like Reddit's. Want me to swap in that
  #    approach, or stick with the windowed sort?"
  #
  # The formula, decoded:
  #   log(upvotes + 1)   →  log smooths big numbers; 100 vs 10,000 isn't
  #                         100× the user attention. Reddit does this too.
  #   +1 inside the log  →  handles upvotes==0 cleanly:
  #                         log(0) = -Infinity, but log(1) = 0. Safe floor.
  #   age_hours / 12     →  every 12 hours of age subtracts 1 from the score
  #                         (tune this constant to taste)
  #   [...,1.0].max      →  guards against zero/negative ages → infinite scores
  #
  # `self.hot_score` (class method) because it doesn't depend on instance state.
  def self.hot_score(post, now)
    age_hours = [((now - post.created_at) / 3600.0), 1.0].max
    Math.log([post.upvotes, 1].max + 1) - (age_hours / 12.0)
  end
end

    Ruby skeleton — the universal starter

    Memorize this. 90% of pairing problems start as some variation of it. Type it from blank twice tonight.

    # 1. Domain object — Struct gives you ==, hash, to_s, accessors for free
Job = Struct.new(:id, :title, :employer, :skills, keyword_init: true)

# 2. Container — Hash keyed by id is the workhorse
class JobBoard
  def initialize
    @jobs = {}
  end

  # --- CRUD ---
  def post(job)
    raise ArgumentError, "duplicate job id: #{job.id}" if @jobs.key?(job.id)
    @jobs[job.id] = job
  end

  def get(id)
    @jobs.fetch(id) { raise KeyError, "no such job: #{id}" }
  end

  def list
    @jobs.values
  end

  def update(id, **attrs)
    job = get(id)
    attrs.each { |k, v| job[k] = v }   # Struct supports []= by symbol
    job
  end

  def delete(id)
    @jobs.delete(id) || (raise KeyError, "no such job: #{id}")
  end

  # --- domain op (the "interesting" method) ---
  def search(skill)
    @jobs.values.select { |j| j.skills.include?(skill) }
  end
end

# 3. Inline asserts after each method — run constantly
board = JobBoard.new
board.post(Job.new(id: "j1", title: "SWE Intern", employer: "Acme", skills: %w[ruby rails]))
raise "fail" unless board.get("j1").title == "SWE Intern"
raise "fail" unless board.search("ruby").length == 1
raise "fail" unless board.search("rust").empty?
puts "ok — part 1"

# 4. Volunteer edge cases out loud
begin
  board.get("nope")
  raise "should have raised"
rescue KeyError => e
  puts "ok — unknown id raises KeyError: #{e.message}"
end

    In-memory CRUD pattern (most common problem shape)

    If the problem is "build a system that does X, Y, Z" — it's almost always this shape:

    1. Domain object — Struct.new(:fields, keyword_init: true)
    2. Container class with a Hash keyed by id (@things = {})
    3. Public methods: create / get / list / update / delete — plus the domain-specific ops
    4. Typed errors: ArgumentError / KeyError with "message: #{id}"
    5. Inline raise "fail" unless ... + puts "ok" after each method

    Strategy / dispatch-table pattern (cleans up Part 2 nicely)

    When Part 2 introduces "different kinds of X" (different job types, different application states, different match algorithms) — don't write a long if/elif. Use a dispatch dict.

    # BAD — case/when grows with every new type
def rank(job_type, candidate, job)
  case job_type
  when :internship then # ...
  when :full_time  then # ...
  when :part_time  then # ...
  end
end

# GOOD — dispatch table, easy to extend, easy to test each ranker alone
RANKERS = {
  internship: ->(c, j) { c.skills.count { |s| j.skills.include?(s) } * 2 },
  full_time:  ->(c, j) { c.years_exp * j.weight + c.skills.size },
  part_time:  ->(c, j) { c.availability_score },
}.freeze

def rank(job_type, candidate, job)
  ranker = RANKERS.fetch(job_type) { raise ArgumentError, "unknown job type: #{job_type}" }
  ranker.call(candidate, job)
end

    Questions to Ask have 3 ready per round — they signal as much as your answers

    The rule

    Pick questions that match the interviewer. Ask a hiring manager about org/team strategy; ask an IC about day-to-day craft; ask leadership about company direction.

    Don't ask anything you could have Googled. The questions are scored too.

    For the hiring manager (leadership round)

    • "How do you see this role contributing to the AI pivot — is it core platform work, or AI-adjacent product work?"
    • "What's the team's most ambitious goal over the next two quarters? What's most likely to slip?"
    • "Six months in, what does a successful person in this role have on their reflection? What does an unsuccessful one have stuck?"
    • "How do you personally balance the 'Olympic pace' with sustainable engineering? Where do you draw the line?"
    • "What's the strongest critique someone could fairly make of your team's engineering practices right now? What are you doing about it?"
    • "How does decision-making work between engineering and product here? Who owns roadmap?"
    • "What's the most interesting hiring or organizational decision you've made in the last year?"

    For a senior engineer / pair-programming interviewer

    • "What's the biggest piece of technical debt the team has right now? How are you approaching it?"
    • "How is the monolith-to-services migration going? Where in the codebase do you still feel the friction?"
    • "What's a recent technical decision your team got wrong and changed?"
    • "How do you actually pair-program in your team — formally scheduled, ad-hoc, on certain kinds of work?"
    • "What's the on-call experience like? How healthy is it?"
    • "What does the gap between a senior and a staff engineer look like on your team?"
    • "What's the most fun thing you've shipped in the last 6 months?"

    Strategic / company-level (any interviewer can field)

    • "With Handshake AI growing as fast as it is, how is engineering being structured around it? Separate org, or shared platform?"
    • "The Taro acquisition — how is the engineer community being woven into Handshake AI? What does that look like 6 months in?"
    • "How is the original early-career platform evolving alongside the AI business? What's the relationship between them?"
    • "What's the biggest cultural shift you've seen since the AI pivot started?"

    The closing question (very effective)

    ALWAYS GOOD CLOSER

    "Based on what we discussed, do you have any hesitations about my fit for this role? I'd love a chance to address them before we wrap up."

    Signals confidence + self-awareness + coachability. Almost no candidates do this and interviewers reliably remember it.

    Questions NOT to ask

    • "What does Handshake do?" / "Who are your competitors?" — Googleable.
    • "How much vacation time do I get?" — Save for recruiter / offer.
    • "Is it remote?" — Same.
    • "How long until I get promoted to staff?" — Reads as transactional.
    • "Do you have layoffs planned?" — Even if you want to know, ask diplomatically: "How has the team been impacted by the strategic shifts?"
    • Anything yes/no — make them open-ended so you actually learn something.
