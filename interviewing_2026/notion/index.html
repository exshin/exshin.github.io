    ★ Text Editor: insert / delete / undo / redo (most consistently reported)

    Prompt: Implement a text document supporting insert(chars) (append to end), delete(k) (remove up to k chars from the end), undo() (undo the last effective operation), and redo() (re-apply the last undone op). Any new operation clears the redo stack.

    Often handed to you as a stubbed Operation base class with InsertAtEndOperation / DeleteFromEndOperation subclasses to complete. The key insight they want: each operation knows how to apply itself AND how to roll itself back (its inverse) — that's the Command pattern, and it makes undo/redo uniform.

    Talk-track (say this as you start)

    1. Clarify: "Does delete(k) remove from the end or at a cursor? If k > length, remove all? Is undo per-operation or per-character?"
    2. Name the model: "I'll use the Command pattern — one class per operation that can apply and rollback. Two stacks: undo and redo. New action clears redo. That keeps undo/redo symmetric and trivial to extend with new op types."
    3. Build incrementally: insert + delete first, run it, then undo, then redo.
    4. Trace the example. State complexity: each op O(1) amortized for append; delete O(k) to capture removed text. Offer follow-ups.

    Reference solution — Command pattern (the answer they want)

    # A text document with full undo/redo.
# Core idea: every edit is an Operation object that knows how to
# apply itself AND how to undo itself (its inverse). Undo/redo is
# then just popping ops off two stacks and calling apply/rollback.
class TextDocument
  attr_reader :text

  def initialize
    @text = ""        # current document contents
    @undo = []        # ops we can undo (newest on top)
    @redo = []        # ops we can redo
  end

  # ---------- public API ----------
  def insert(chars)
    apply(InsertOp.new(chars))
  end

  def delete(k)
    apply(DeleteOp.new(k))
  end

  def undo
    return if @undo.empty?
    op = @undo.pop
    op.rollback(self)   # restore text to before this op
    @redo.push(op)      # it can now be redone
  end

  def redo
    return if @redo.empty?
    op = @redo.pop
    op.apply(self)      # re-run it
    @undo.push(op)      # it can be undone again
  end

  # ---------- primitives the operations use ----------
  def append(str)
    @text += str
  end

  # Remove up to n chars from the end and RETURN what was removed,
  # so a delete can undo itself by putting the text back.
  def truncate(n)
    n = [n, @text.length].min
    removed = @text[@text.length - n, n]
    @text = @text[0, @text.length - n]
    removed
  end

  private

  def apply(op)
    op.apply(self)
    @undo.push(op)
    @redo.clear         # a brand-new action invalidates the redo history
  end
end

# Each operation captures whatever it needs to reverse itself.
class InsertOp
  def initialize(chars)
    @chars = chars
  end

  def apply(doc)
    doc.append(@chars)
  end

  def rollback(doc)
    doc.truncate(@chars.length)   # remove exactly what we added
  end
end

class DeleteOp
  def initialize(k)
    @k = k
    @removed = nil               # captured at apply-time; needed to undo
  end

  def apply(doc)
    @removed = doc.truncate(@k)  # remember what we removed
  end

  def rollback(doc)
    doc.append(@removed)         # put it back, exactly
  end
end

    doc = TextDocument.new
doc.insert("Hello")    # "Hello"
doc.insert(" World")   # "Hello World"
doc.delete(6)          # "Hello"          (removed " World")
doc.undo               # "Hello World"    (DeleteOp re-appends " World")

    Simpler alternative — snapshot stack (the brute force; name it, don't lead with it)

    Push the whole string before each edit. Trivial to write, but O(n) memory per edit. Good to mention as the naive baseline, then contrast with the command pattern's O(delta) memory.

    class SnapshotDoc
  def initialize
    @text = ""
    @undo = []
    @redo = []
  end
  attr_reader :text

  def insert(chars); checkpoint; @text += chars; end
  def delete(k);     checkpoint; @text = @text[0, [@text.length - k, 0].max]; end

  def undo
    return if @undo.empty?
    @redo.push(@text)
    @text = @undo.pop
  end

  def redo
    return if @redo.empty?
    @undo.push(@text)
    @text = @redo.pop
  end

  private
  def checkpoint
    @undo.push(@text)   # save full state before mutating
    @redo.clear
  end
end

    Follow-ups they may add live (be ready)

    • insert-at-position / delete-range: add InsertAtOp(pos, chars) — apply splices in, rollback splices out. The command pattern absorbs this with one new class; the snapshot version needs no change. Lead with "this is exactly why I chose the Command pattern."
    • Batch / transaction: a CompositeOp holding a list of ops; apply runs them in order, rollback runs the inverses in reverse order. One undo reverts the batch.
    • Bounded history: cap the undo stack (e.g. last 100) — drop the oldest from the bottom.
    • Cursor / selection: track a cursor index; operations move it; undo restores it.
    • Collaborative editing: "this is where OT or CRDTs come in — concurrent edits need transformation, not a single linear stack." (See Notion Model tab.)

    Worked solutions for the follow-ups (click to expand)

    Each builds on the base TextDocument above. Notice the pattern: new behavior = a new operation class (sometimes a new primitive), and the undo/redo machinery never changes. That invariance is the point you're selling.

    1 · Insert-at-position & delete-range

    Add two primitives that work at an arbitrary index, plus one op class each. The two stacks are untouched.

    # --- add these two primitives inside TextDocument ---

# Insert `str` so it begins at index `pos`.
def write_at(pos, str)
  @text = @text[0, pos] + str + @text[pos, @text.length]
end

# Remove `len` chars starting at `pos`; RETURN what was removed (for undo).
def erase(pos, len)
  removed = @text[pos, len] || ""
  @text = @text[0, pos] + @text[pos + len, @text.length]
  removed
end

# --- and these one-line public methods inside TextDocument ---
def insert_at(pos, chars); apply(InsertAtOp.new(pos, chars)); end
def delete_range(pos, len); apply(DeleteRangeOp.new(pos, len)); end

# --- two new operation classes (same apply/rollback shape as before) ---
class InsertAtOp
  def initialize(pos, chars)
    @pos = pos
    @chars = chars
  end

  def apply(doc)
    doc.write_at(@pos, @chars)
  end

  def rollback(doc)
    doc.erase(@pos, @chars.length)   # remove exactly what we inserted
  end
end

class DeleteRangeOp
  def initialize(pos, len)
    @pos = pos
    @len = len
    @removed = nil                   # captured at apply-time
  end

  def apply(doc)
    @removed = doc.erase(@pos, @len)
  end

  def rollback(doc)
    doc.write_at(@pos, @removed)     # put the removed slice back
  end
end

# doc.insert("Hello World"); doc.insert_at(5, ",")  => "Hello, World"
# doc.undo                                           => "Hello World"
    2 · Batch / transaction (one undo reverts many edits)

    A composite is itself an operation that holds child ops: apply them in order, roll them back in reverse. It pushes as a single entry, so one undo reverts the whole batch.

    # An operation made of other operations (the Composite pattern).
class CompositeOp
  def initialize(ops)
    @ops = ops
  end

  def apply(doc)
    @ops.each { |op| op.apply(doc) }            # forward order
  end

  def rollback(doc)
    @ops.reverse_each { |op| op.rollback(doc) } # reverse order to undo cleanly
  end
end

# public method inside TextDocument:
def batch(*ops)
  apply(CompositeOp.new(ops))
end

# doc.batch(InsertOp.new("Hello"), InsertOp.new(" World"))  => "Hello World"
# doc.undo   # reverts BOTH inserts at once                  => ""
    3 · Bounded history (cap the undo stack)

    One line in apply: after pushing, drop the oldest if we're over the cap. Those edits simply become non-undoable — correct behavior.

    class TextDocument
  MAX_HISTORY = 100

  # ... rest unchanged ...

  private

  def apply(op)
    op.apply(self)
    @undo.push(op)
    @undo.shift if @undo.length > MAX_HISTORY   # forget the oldest edit
    @redo.clear
  end
end
    4 · Cursor / selection

    Track a cursor index; insert/delete happen there, and each op restores the cursor on undo. Here the primitives keep the cursor consistent, so the ops stay tiny. (delete is a backspace: it removes the chars before the cursor.)

    class CursorDocument
  attr_reader :text, :cursor

  def initialize
    @text = ""
    @cursor = 0           # where the next edit happens
    @undo = []
    @redo = []
  end

  def insert(chars); run(CursorInsert.new(@cursor, chars)); end
  def delete(k);     run(CursorDelete.new(@cursor, k));     end
  def move(pos);     @cursor = pos;                          end

  def undo
    return if @undo.empty?
    op = @undo.pop
    op.rollback(self)
    @redo.push(op)
  end

  def redo
    return if @redo.empty?
    op = @redo.pop
    op.apply(self)
    @undo.push(op)
  end

  # Primitives ALSO move the cursor, so undo/redo restore it for free:
  # after writing, the cursor sits just past the new text; after erasing,
  # it sits where the erase began.
  def write(pos, str)
    @text = @text[0, pos] + str + @text[pos, @text.length]
    @cursor = pos + str.length
  end

  def erase(pos, len)
    removed = @text[pos, len] || ""
    @text = @text[0, pos] + @text[pos + len, @text.length]
    @cursor = pos
    removed
  end

  private

  def run(op)
    op.apply(self)
    @undo.push(op)
    @redo.clear
  end
end

class CursorInsert
  def initialize(cursor, chars)
    @cursor = cursor
    @chars = chars
  end

  def apply(doc)
    doc.write(@cursor, @chars)        # leaves cursor after the insert
  end

  def rollback(doc)
    doc.erase(@cursor, @chars.length) # leaves cursor back at the insert point
  end
end

class CursorDelete
  def initialize(cursor, k)
    @k = [k, cursor].min              # can't delete past the start
    @start = cursor - @k
    @removed = nil
  end

  def apply(doc)
    @removed = doc.erase(@start, @k)
  end

  def rollback(doc)
    doc.write(@start, @removed)       # restores text AND cursor
  end
end

# c.insert("Hello"); c.insert(" World")  # "Hello World", cursor 11
# c.move(5); c.insert(",")               # "Hello, World", cursor 6
# c.undo                                  # "Hello World", cursor 5
    5 · Collaborative editing — illustrative operational transform

    Be honest here: a single undo/redo stack can't handle two people editing at once — concurrent ops must be transformed against each other so each user's intent survives. Real OT/CRDT also handles deletes, tie-breaking, and every op-pair; this snippet shows only the core idea (insert-vs-insert) to prove you understand the shape, not a production algorithm.

    # Two users edit the SAME original doc concurrently. Op = {pos:, text:}.
# After applying `a`, transform `b` so it still lands where the user meant.
def transform_insert(a, b)
  if a[:pos] <= b[:pos]
    # `a` inserted to the left of `b`, so everything at/after b shifted right.
    { pos: b[:pos] + a[:text].length, text: b[:text] }
  else
    b                                  # `a` was to the right -> `b` unaffected
  end
end

def apply_insert(text, op)
  text[0, op[:pos]] + op[:text] + text[op[:pos], text.length]
end

# base = "abc"
# a = {pos: 0, text: "X"}   # user A: prepend X
# b = {pos: 2, text: "Y"}   # user B: insert Y before "c"
# after_a = apply_insert("abc", a)              # "Xabc"
# b2 = transform_insert(a, b)                   # {pos: 3, text: "Y"}
# apply_insert(after_a, b2)                     # "XabYc"  (both intents kept)

    Sliding Window Median (2026 companion · LC 480)

    Prompt: Given an array nums and a window size k, return the median of every window as it slides one step at a time. Median of an even-sized window is the average of the two middles.

    This is the one genuinely algorithmic prompt in Notion's recent cluster. Lead with the clean sorted-window approach (maintain the window sorted; binary-search to remove the outgoing element and insert the incoming one) — easy to narrate and obviously correct. Then name the optimization: two heaps (a max-heap of the lower half, min-heap of the upper half) with lazy deletion gives O(log k) per step vs. O(k) for the sorted list. State that trade-off out loud even if you code the simpler one.

    Sorted-window solution (clean, O(k) per step — code this)

    # Median of a SORTED array: the middle for odd length,
# the average of the two middles for even length.
def median_of(sorted)
  mid = sorted.length / 2
  if sorted.length.odd?
    sorted[mid].to_f
  else
    (sorted[mid - 1] + sorted[mid]) / 2.0
  end
end

# Keep the current window SORTED. Each step: drop the element leaving
# the window, insert the one entering it, both via binary search.
def median_sliding_window(nums, k)
  window = nums[0, k].sort
  medians = [median_of(window)]

  (k...nums.length).each do |i|
    # remove the outgoing value nums[i-k] (bsearch finds its leftmost copy)
    out = nums[i - k]
    window.delete_at(window.bsearch_index { |x| x >= out })

    # insert the incoming value nums[i] at its sorted position
    inp = nums[i]
    pos = window.bsearch_index { |x| x >= inp } || window.length
    window.insert(pos, inp)

    medians << median_of(window)
  end
  medians
end

# median_sliding_window([1,3,-1,-3,5,3,6,7], 3) => [1.0,-1.0,-1.0,3.0,5.0,6.0]

    The optimization to name (don't have to code it)

    Two heaps: a max-heap (lo) for the smaller half and a min-heap (hi) for the larger half, kept balanced so the median is the top of lo (odd) or the average of both tops (even). The hard part is removing the element leaving the window — heaps don't support arbitrary delete, so you use lazy deletion: keep a hash of "to-delete" counts and purge entries only when they surface at a heap top. That's the O(log k) answer; say it, then decide with the interviewer whether it's worth coding in the time left.

    Task Queue / scheduler modifications (2026 companion)

    Prompt (representative — the exact spec varies by report): Implement a task queue / scheduler with:

    • add(task_id, priority) — enqueue a task with a priority (higher = more urgent).
    • next_task — remove and return the most urgent task.
    • update_priority(task_id, p) — change a queued task's priority.
    • cancel(task_id) — remove a task so it never runs.

    Ties (equal priority) break by insertion order — earliest task first.

    Reading the problem: where's the difficulty?

    If the only operations were add and next_task, a binary heap (priority queue) would be the textbook answer: O(log n) for both. But the twist is update_priority and cancel — and a plain heap can't efficiently change or remove an element in the middle; it only knows about its top.

    So the design question is: how do I support fast update/cancel? The simplest correct structure is a hash from task_id to its priority. Updating or cancelling a task is then just a hash write or delete — O(1). The cost moves to next_task, which scans the hash to find the most urgent task — O(n). For an interview that's a perfectly good first answer; you then name the optimization rather than rushing to it.

    The data we keep

    • @tasks — a hash task_id => { priority:, order: }. Each task is ONE entry holding both its priority and its arrival step. This IS the queue.
    • @next_order — a counter that goes up by one on each new task, so arrival steps are unique and increasing (smaller = added earlier = wins ties).

    Keeping priority and order together makes each task a single source of truth: add/update/cancel each touch exactly one entry, so the two values can never drift out of sync. (An earlier draft used two parallel hashes — cleaner to write on a whiteboard, but then cancel has to remember to delete from both, which is a desync bug waiting to happen.)

    The solution (every method short and obvious)

    class TaskQueue
  def initialize
    @tasks = {}        # task_id => { priority:, order: }  (this is the queue)
    @next_order = 0    # increases by 1 per task, so earlier = smaller number
  end

  # Add a task, or re-add one. Re-adding updates the priority but KEEPS the
  # original arrival order, so a task that's been waiting keeps its place.
  def add(task_id, priority)
    if @tasks.key?(task_id)
      @tasks[task_id][:priority] = priority
    else
      @tasks[task_id] = { priority: priority, order: @next_order }
      @next_order += 1
    end
  end

  # Change a task's priority. If we don't know the task (never added, or
  # already run/cancelled), do nothing.
  def update_priority(task_id, priority)
    return unless @tasks.key?(task_id)
    @tasks[task_id][:priority] = priority
  end

  # Remove a task entirely so it will never run. One entry, one delete.
  def cancel(task_id)
    @tasks.delete(task_id)
  end

  # Remove and return the most urgent task:
  #   1. the highest priority, and
  #   2. among ties, the one that was added earliest.
  def next_task
    return nil if @tasks.empty?

    highest    = @tasks.values.map { |t| t[:priority] }.max         # best priority present
    candidates = @tasks.keys.select { |id| @tasks[id][:priority] == highest }
    winner     = candidates.min_by { |id| @tasks[id][:order] }      # earliest arrival wins

    cancel(winner)   # dequeue it
    winner
  end
end

    Task = Struct.new(:priority, :order)

    Trace it out loud

    q = TaskQueue.new
q.add("a", 1)         # {a:{priority:1, order:0}}
q.add("b", 5)         # {a:{1,0}, b:{5,1}}
q.add("c", 3)         # {a:{1,0}, b:{5,1}, c:{3,2}}

q.next_task                  # highest=5    -> "b"   (now removed)
q.update_priority("a", 10)   # a jumps the line: a now {priority:10, order:0}
q.next_task                  # highest=10   -> "a"

q.add("d", 3)         # {c:{3,2}, d:{3,3}}   order c=2 < d=3
q.add("e", 3)         # {c:{3,2}, d:{3,3}, e:{3,4}}
q.next_task           # three-way tie at 3 -> earliest is "c"
q.cancel("d")         # remove d before it ever runs
q.next_task           # "e"   (d was cancelled, skipped entirely)
q.next_task           # nil   (queue empty)

    Complexity & the upgrade to name

    • add, update_priority, cancel: O(1).
    • next_task: O(n) — it scans every queued task. Fine for interview scale; say so.
    • Upgrade (state it, decide together whether to code it): use a binary heap ordered by (priority, arrival) to make next_task O(log n). To handle update/cancel, use lazy deletion: keep the @tasks hash as the source of truth, push a new heap entry on update, and when you pop, skip any entry whose priority no longer matches the hash (or whose task was cancelled). You trade a little memory to avoid an expensive arbitrary-delete.

    Follow-ups: scheduled/delayed tasks (only runnable after time T — add a ready_at and filter next_task to ready tasks, or use a time-ordered structure); dependencies between tasks (becomes the DAG / topological-sort problem below); fairness / round-robin so low-priority tasks don't starve; persisting the queue so it survives a restart.

    Binary search (+ lower bound)

    Prompt: Given a sorted array, return the index of target (or -1 if absent). Follow-up: return the position where target would be inserted to keep the array sorted (the "lower bound").

    A fundamental that shows up directly or hidden inside other problems (it's what the Sliding Window Median solution uses to place elements). The signal is off-by-one discipline: be deliberate about inclusive vs. exclusive bounds and why the loop terminates. Ruby has Array#bsearch_index built in, but interviewers usually want you to write it by hand — mention the builtin, then write it.

    Classic binary search — find the target, or -1

    # Inclusive bounds: lo and hi both point at real, searchable indices.
# Loop runs while the range [lo..hi] is non-empty, i.e. lo <= hi.
def binary_search(nums, target)
  lo = 0
  hi = nums.length - 1            # hi is a valid index (inclusive)

  while lo <= hi
    mid = (lo + hi) / 2           # integer division; Ruby ints don't overflow
    if nums[mid] == target
      return mid                  # found it
    elsif nums[mid] < target
      lo = mid + 1                # target must be in the right half
    else
      hi = mid - 1               # target must be in the left half
    end
  end

  -1                             # range emptied out -> not present
end

# binary_search([1,3,5,7,9], 7) => 3
# binary_search([1,3,5,7,9], 4) => -1

    Lower bound — first index whose value is >= target (the insertion point)

    The more reusable variant: it answers "where would target go?" and finds the leftmost match when there are duplicates. Note the bounds differ on purpose — hi starts at length (one past the end) so the answer can be "append here," and the loop is lo < hi with hi = mid (never mid - 1) so we never skip the candidate.

    # Exclusive upper bound: hi can equal length (a valid "insert at end" answer).
def lower_bound(nums, target)
  lo = 0
  hi = nums.length               # NOT length - 1

  while lo < hi
    mid = (lo + hi) / 2
    if nums[mid] < target
      lo = mid + 1               # everything up to mid is too small
    else
      hi = mid                   # mid might be the answer -> keep it in range
    end
  end

  lo                             # == hi: the first index with value >= target
end

# lower_bound([1,3,5,7,9], 4)  => 2   (5 is the first value >= 4)
# lower_bound([1,3,5,7,9], 10) => 5   (past the end: insert at the end)
# lower_bound([2,2,2], 2)      => 0   (leftmost of the duplicates)

# Ruby builtin equivalent (find-minimum mode):
# nums.bsearch_index { |x| x >= target }   # nil if none qualify

    O(log n) time, O(1) space. Follow-ups: upper bound (first index > target → count occurrences as upper - lower), search a rotated sorted array, find first/last position of a target, search for a boundary/condition rather than a value.

    In-memory store + transactions (the "build a few APIs" round)

    Prompt: Build a key-value store with get, set, delete, and nested transactions: begin, commit, rollback. A common phone-screen framing of "implement a tiny server / a few APIs — an in-memory store is fine."

    Key insight: a stack of transaction layers. Reads search top-down; writes go to the top layer; commit merges down; rollback discards the top. Use a tombstone marker so a delete inside a transaction can shadow a committed value.

    class KVStore
  TOMBSTONE = :__deleted__   # marks "deleted within a transaction"

  def initialize
    @base = {}     # committed data
    @txns = []     # stack of pending layers (each a Hash of writes)
  end

  # Read: newest layer wins; fall through to base.
  def get(key)
    @txns.reverse_each do |layer|
      next unless layer.key?(key)
      val = layer[key]
      return val == TOMBSTONE ? nil : val
    end
    @base[key]
  end

  def set(key, value)
    layer = @txns.last || @base
    layer[key] = value
  end

  def delete(key)
    if @txns.empty?
      @base.delete(key)
    else
      @txns.last[key] = TOMBSTONE   # shadow lower layers without touching them
    end
  end

  def begin
    @txns.push({})
  end

  # Merge the top layer into the one below (or base).
  def commit
    raise "no open transaction" if @txns.empty?
    layer  = @txns.pop
    target = @txns.last || @base
    layer.each do |k, v|
      if v == TOMBSTONE
        @txns.empty? ? @base.delete(k) : (target[k] = TOMBSTONE)
      else
        target[k] = v
      end
    end
  end

  def rollback
    raise "no open transaction" if @txns.empty?
    @txns.pop   # throw away pending writes
  end
end

kv.rollback

kv.begin
kv.delete("a")
kv.get("a")     # => nil (tombstone)

kv.get("a")     # => nil (committed)

    Follow-ups: TTL/expiry, list keys by prefix, snapshot/restore, persistence to a log (append-only), concurrent access (a lock per key).

    Block tree — Notion's actual data model

    Prompt (Notion-flavored): A page is a tree of blocks; each block has an id, text, and ordered children. Implement: render to indented text, move a block under a new parent, and indent/outdent a block.

    In Notion, indentation is structural — it reflects the parent/child render tree, not styling. Many "manipulate this nested thing" prompts are this in disguise. Reach for recursion + a parent/children map.

    # A block holds its own children (the render tree).
class Block
  attr_accessor :id, :text, :children
  def initialize(id, text, children = [])
    @id = id
    @text = text
    @children = children
  end
end

# Render the subtree as indented plain text (depth-first).
def render(block, depth = 0)
  lines = ["  " * depth + block.text]
  block.children.each do |child|
    lines.concat(render(child, depth + 1))
  end
  lines
end

# Index every block by id, and remember each block's parent,
# so move/indent operations are O(1) lookups.
def index(root)
  by_id, parent = {}, {}
  stack = [[root, nil]]
  until stack.empty?
    node, par = stack.pop
    by_id[node.id] = node
    parent[node.id] = par
    node.children.each { |c| stack.push([c, node]) }
  end
  [by_id, parent]
end

# Move `block_id` to be the last child of `new_parent_id`.
def move(root, block_id, new_parent_id)
  by_id, parent = index(root)
  node, old_parent = by_id[block_id], parent[block_id]
  old_parent.children.delete(node)               # detach
  by_id[new_parent_id].children.push(node)       # re-attach
end

# Indent: make a block a child of its previous sibling (like Tab in Notion).
def indent(root, block_id)
  _, parent = index(root)
  par = parent[block_id]
  idx = par.children.index { |c| c.id == block_id }
  return if idx.nil? || idx == 0      # nothing to indent under
  node = par.children.delete_at(idx)
  par.children[idx - 1].children.push(node)
end

    Follow-ups: outdent (inverse), serialize to JSON, find a block by id, compute a block's full path, permissions inherited down the tree.

    Nested to-dos list API

    Prompt: Build an in-memory API for a hierarchical to-do list — todos can nest under other todos to any depth. Support add(text, parent_id = nil), set_done(id), remove(id), and render (the indented list with checkboxes).

    This is the "build a few APIs over an in-memory store" round wearing a Notion costume — a to-do block in Notion is a node in a nested tree. The whole problem is a tree keyed by id, so reach for the same toolkit as the block tree: a hash of id => node with a children list and a parent pointer.

    Two design questions to raise out loud

    • Does completing a parent complete its subtasks? Usually yes — confirm it, then cascade. (Shown below.)
    • Does deleting a todo delete its subtree? Almost always yes — otherwise the children are orphaned. Cascade the delete.

    Trick that simplifies everything: a virtual root. Top-level todos are just children of a hidden root node, so "add to the top level" and "add under a parent" are the same code path.

    The solution

    class TodoList
  ROOT = 0   # virtual root: real top-level todos are its children

  def initialize
    @nodes = { ROOT => node("", nil) }   # id => { text:, done:, parent:, children: [] }
    @next_id = 1
  end

  # Add a todo (optionally under a parent) and return its new id.
  def add(text, parent_id = ROOT)
    id = @next_id
    @next_id += 1
    @nodes[id] = node(text, parent_id)
    @nodes[parent_id][:children] << id    # append under the parent (order preserved)
    id
  end

  # Mark a todo done/undone, cascading to all of its subtasks.
  def set_done(id, done = true)
    return unless @nodes.key?(id)
    @nodes[id][:done] = done
    @nodes[id][:children].each { |child| set_done(child, done) }
  end

  # Delete a todo and its entire subtree.
  def remove(id)
    return if id == ROOT || !@nodes.key?(id)   # never delete the virtual root
    parent = @nodes[id][:parent]
    @nodes[parent][:children].delete(id)        # detach from its parent
    delete_subtree(id)                          # free the node and its descendants
  end

  # Render as indented text with checkboxes.
  def render
    lines = []
    @nodes[ROOT][:children].each { |id| render_into(id, 0, lines) }
    lines
  end

  private

  def node(text, parent_id)
    { text: text, done: false, parent: parent_id, children: [] }
  end

  def delete_subtree(id)
    @nodes[id][:children].each { |child| delete_subtree(child) }   # depth-first
    @nodes.delete(id)
  end

  def render_into(id, depth, lines)
    n = @nodes[id]
    box = n[:done] ? "[x]" : "[ ]"
    lines << ("  " * depth) + "#{box} #{n[:text]}"
    n[:children].each { |child| render_into(child, depth + 1, lines) }
  end
end

Every one of these is a small tree walk over the same @nodes hash.

    Table group-by aggregation (Notion DB rollups)

    Prompt: Given rows of { key:, value: }, group by key and aggregate (sum / count / avg / min / max). This is the "implement table aggregation" warm-up — Notion databases are tables with rollups.

    # Simple: sum values per group.
def sum_by_key(rows)
  totals = Hash.new(0)
  rows.each { |row| totals[row[:key]] += row[:value] }
  totals
end

# General: pluggable aggregations in one pass over the data.
def aggregate(rows)
  acc = Hash.new { |h, k| h[k] = { count: 0, sum: 0, min: nil, max: nil } }
  rows.each do |row|
    k, v = row[:key], row[:value]
    a = acc[k]
    a[:count] += 1
    a[:sum]   += v
    a[:min]    = v if a[:min].nil? || v < a[:min]
    a[:max]    = v if a[:max].nil? || v > a[:max]
  end
  acc.transform_values { |a| a.merge(avg: a[:sum].to_f / a[:count]) }
end

# rows = [{key:"a",value:3},{key:"b",value:5},{key:"a",value:7}]
# sum_by_key(rows) => {"a"=>10, "b"=>5}

    Follow-ups: group by multiple columns, filter then aggregate, sort groups, streaming rows (keep it one-pass).

    File system / nested paths

    Prompt: Implement an in-memory file system: mkdir(path), add_file(path, content), ls(path). Model it as a nested hash (a trie of path segments) — same tree skill as the block model.

    class FileSystem
  def initialize
    @root = {}            # nested Hash; files stored under a :__file__ key
  end

  def mkdir(path)
    walk(split(path), create: true)
  end

  def add_file(path, content)
    parts = split(path)
    dir = walk(parts[0...-1], create: true)
    dir[parts.last] = { __file__: content }
  end

  # Return sorted names in a directory; if path is a file, return [basename].
  def ls(path)
    node = walk(split(path), create: false)
    return [split(path).last] if node.key?(:__file__)
    node.keys.reject { |k| k == :__file__ }.sort
  end

  private

  def split(path)
    path.split("/").reject(&:empty?)
  end

  def walk(parts, create:)
    node = @root
    parts.each do |part|
      node[part] = {} if !node.key?(part) && create
      node = node[part] or raise "no such path"
    end
    node
  end
end

    DAG task graph (topological order)

    Prompt: Build a graph of tasks with dependencies; return a valid execution order (or detect a cycle). Kahn's algorithm (BFS on in-degrees) — clean and cycle-detecting.

    class TaskGraph
  def initialize
    @deps = Hash.new { |h, k| h[k] = [] }   # task => tasks that depend on it
    @indeg = Hash.new(0)                     # task => number of prerequisites
    @tasks = []
  end

  def add_task(id)
    return if @indeg.key?(id)
    @tasks << id
    @indeg[id] = 0
  end

  # `task` requires `prereq` to run first.
  def add_dependency(task, prereq)
    add_task(task); add_task(prereq)
    @deps[prereq] << task
    @indeg[task] += 1
  end

  # Topological order, or nil if there's a cycle.
  def order
    indeg = @indeg.dup
    queue = @tasks.select { |t| indeg[t] == 0 }
    result = []
    until queue.empty?
      t = queue.shift
      result << t
      @deps[t].each do |nxt|
        indeg[nxt] -= 1
        queue << nxt if indeg[nxt] == 0
      end
    end
    result.length == @tasks.length ? result : nil   # nil => cycle
  end
end

    LRU cache

    Prompt: get/put in O(1) with a capacity; evict least-recently-used. Ruby's Hash preserves insertion order, so delete+reinsert moves a key to "most recent" — no manual linked list needed (mention you'd use a doubly-linked list + hash in a language without ordered maps).

    class LRUCache
  def initialize(capacity)
    @cap = capacity
    @h = {}   # insertion-ordered: front = oldest, back = newest
  end

  def get(key)
    return nil unless @h.key?(key)
    val = @h.delete(key)
    @h[key] = val            # reinsert -> now most-recent
    val
  end

  def put(key, value)
    @h.delete(key)           # remove if present so it re-inserts at the back
    @h[key] = value
    @h.delete(@h.keys.first) if @h.size > @cap   # evict oldest
  end
end

    Rate limiter / logger (sliding window)

    Prompt: "Should this message print?" — allow a key at most once per N seconds (logger), or N times per window (rate limiter). Track timestamps per key.

    # Logger: allow a message at most once every `interval` seconds.
class Logger
  def initialize(interval = 10)
    @interval = interval
    @last = {}                      # message => last-allowed timestamp
  end

  def should_print?(timestamp, message)
    if !@last.key?(message) || timestamp - @last[message] >= @interval
      @last[message] = timestamp
      true
    else
      false
    end
  end
end

# Rate limiter: at most `limit` events per `window` seconds, per key.
class RateLimiter
  def initialize(limit, window)
    @limit = limit
    @window = window
    @events = Hash.new { |h, k| h[k] = [] }   # key => recent timestamps
  end

  def allow?(key, now)
    q = @events[key]
    q.shift while !q.empty? && q.first <= now - @window   # drop stale
    return false if q.length >= @limit
    q << now
    true
  end
end

    String parsing / tokenizer

    Prompt: Parse a small format — e.g. tokenize markdown-ish text into runs, or expand "3[ab]2[c]". String manipulation shows up framed as "parse this document/input." Lead with a single left-to-right pass and a stack for nesting.

    # Decode "3[ab]2[c]" => "ababab" + "cc" = "abababcc"
# Stack pattern: push context on '[', resolve on ']'.
def decode(s)
  count_stack = []
  str_stack  = []
  current = ""
  num = 0
  s.each_char do |ch|
    if ch =~ /\d/
      num = num * 10 + ch.to_i
    elsif ch == "["
      count_stack.push(num)
      str_stack.push(current)
      num = 0
      current = ""
    elsif ch == "]"
      repeat = count_stack.pop
      current = str_stack.pop + current * repeat
    else
      current += ch
    end
  end
  current
end

# decode("3[ab]2[c]") => "abababcc"

    Playbook how to run a practical coding round

    The 60-minute shape

    TimePhaseWhat you're doing
    0–3 minClarifyRestate the feature, ask 2–3 sharp questions (edge cases, scope, exact behavior).
    3–8 minModelDefine the interface + data structure OUT LOUD. Get a nod before coding. This is the strongest signal.
    8–35 minBuild incrementallySmallest working slice first, run it, then add the next piece. Test after each.

    The practical-round framework (use every time)

    1. Restate the feature in your own words — confirm you're solving the right thing.
    2. Clarify behavior and edges: "delete(k) with k > length?", "is undo per-op or per-char?", "do operations mutate or return new state?"
    3. Propose the model — name the data structure and the interface. "I'll use X because it makes operation Y cheap and is easy to extend." Get a nod.
    4. Stub the API first — write the method signatures, then fill them in. Shows structure.
    5. Build the simplest slice, then run it. Don't write everything before executing.
    6. Narrate intent, not syntax — explain non-obvious choices only.
    7. Trace an example + state complexity unprompted.
    8. End with: "Want me to add tests, handle more edge cases, or extend it?"

    ★ The AI-tool round (new in 2026)

    Notion has reportedly added an AI-enabled interview that expects fluency with tools like Cursor and Claude Code.

    This may be a separate round or an option within the coding round — confirm with your recruiter which format you'll get. If it's AI-enabled, the thing being measured changes: it's no longer "can you write the code from memory," it's "do I trust your judgment when you're driving an AI." Notion eats its own dog food here — they recently opened the workspace to Claude Code / Cursor / Codex as native agents, so they care that you use these tools well.

    HOW TO SCORE WELL WITH AI IN THE ROOM

    • Drive, don't defer. You own the design and the decisions; the AI is a fast pair, not the engineer. State your plan before you prompt, then steer it toward that plan.
    • Treat every output as a first draft. Blind acceptance is the fastest way to fail. Read what it produced, say what's right and what's suspect, and fix or re-prompt. The critique is the signal.
    • Narrate the prompt and the why. "I'm asking it to scaffold the Operation classes so I can focus on the undo/redo logic" shows intentional use, not vibe-coding.
    • Verify, out loud. Run it, add asserts, trace an example — same testing discipline as a normal round, but now you're also auditing the AI. Catching its bug is a big green flag.
    • Keep the abstraction yours. Don't let AI sprawl the design. "Do less, but better" still applies — prune what it over-generates.
    • Know the tool's moves. Be comfortable with inline edits, multi-file context, asking for a diff, and rejecting a suggestion. Fumbling the UI reads as inexperience.
    • Own mistakes fast. If the AI (or you) shipped a bug, name it and fix it calmly — "would I trust this person when things get messy" is literally what they're scoring.

    AI-ROUND ANTI-PATTERNS

    • Pasting the prompt and accepting whatever comes back without reading it ("prompt jockey").
    • Going silent while the AI works — narrate what you're evaluating.
    • Letting the AI pick the data model for you; you should already know the shape (see Coding tab).
    • Not being able to explain code the AI wrote — if you ship it, you must be able to defend every line.

    Narration that scores (vs. noise)

    SAY THIS

    • "I'll capture the deleted text on the op so undo can restore it."
    • "A stack of layers means commit is just a merge-down — that's why I chose it."
    • "Let me run this before I add undo, to be sure the base works."
    • "That's O(k) for the copy; the rest is O(1)."

    SKIP THIS

    • "Now I'll write a for loop…"
    • Silent typing for a minute.
    • "This is probably wrong but…" (just say what you're testing).
    • Reciting Big-O of every line.

    Testing as you go (a Notion green flag)

    • After each method works, run a quick assertion or print. Their guidance literally values candidates who verify.
    • Keep a scratch driver at the bottom — the trace blocks in the Coding tab are exactly this.
    • Test the happy path, then one edge (empty input, k > length, undo with empty history).
    • When a follow-up lands, add a test for it first — shows you think in behavior.

    Handling the live extension (the real test)

    They WILL add a requirement once it works. This is designed to see if your abstraction bends or breaks.

    • Pause and restate the new requirement. Don't rush.
    • Connect it to your model: "Because each op is its own class, I just add one more type." That sentence is the whole point — your earlier design choice pays off here.
    • If it doesn't fit cleanly, say so honestly and describe the refactor: "My current model assumes append-only; to support insert-at-position I'd change truncate into a splice." Reasoning beats pretending.

    Edge-case checklist (skim before "I'm done")

    • Empty input / empty document
    • Operation when history is empty (undo/redo no-op)
    • k larger than available (clamp, don't crash)
    • Duplicate keys / repeated ids
    • New action clears redo (don't forget!)
    • Mutation vs. fresh copy (aliasing bugs)
    • Order preserved where it matters
    • Off-by-one on ranges/indices

    If you get stuck

    • Re-read the spec aloud. Practical prompts are open — re-confirm the exact behavior.
    • Trace the smallest example by hand on the side.
    • Ask "what's the simplest structure that makes these ops cheap?" — it usually names the answer.
    • Get the naive version working, then say "now let me make it cleaner / faster." Working-then-better always beats stuck-on-perfect.

    Notion Model the data model behind the product

    Why this tab exists

    Your R1 is coding, but many Notion practical problems are "manipulate this nested structure" — which is the block model in disguise. Knowing how Notion is actually built lets you (a) recognize the pattern fast and (b) drop a credible soundbite that signals you understand the product as a system, not just an app.

    The block data model

    • Everything is a block. A paragraph, a heading, a to-do, an image, a page — all blocks. A page is just a block whose children are other blocks.
    • Each block has an id, a type, properties (its content/text), and an ordered list of child block ids.
    • The block's type determines how it renders and how it treats its children.
    • This uniformity is the "LEGO" philosophy: a small set of composable primitives builds docs, wikis, databases, and projects.

    The render tree

    The hierarchy of blocks and the children they render is called the render tree. It isn't a clean textbook tree — different block types render children differently (a toggle hides them, a column lays them side by side) — but for interview purposes, model it as a tree of blocks each holding child ids.

    Practical takeaway: index blocks by id in a hash, keep a parent pointer, and most operations (move, indent, delete-subtree) become O(1) lookups + a tree walk.

    Indentation is structure, not style

    When you indent a block in Notion you're changing the parent/child relationship in the render tree — not adding left-padding. This is the single most "Notion" fact about the data model, and it's exactly the insight the block-tree coding problem is testing. (See Coding tab → Block tree.)

    Lite system-design awareness (if it comes up later)

    TopicThe one thing to say
    Collaborative editingConcurrent edits need OT (operational transform) or CRDTs — transform/merge concurrent ops rather than a single linear undo stack.
    Offline-firstClient holds local state, queues operations, syncs and reconciles when back online. Idempotent ops + version vectors.
    Full-text search across blocksInverted index over block text; update on edit; rank + permission-filter results.
    PermissionsInherited down the block tree from the page/workspace, with overrides — a tree-walk again.
    StorageBlocks in Postgres (a row per block with a parent reference); the tree is reconstructed by parent id.
