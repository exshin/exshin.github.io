    Cheat Sheet scan this between parts

    What they score

    Their own brief, decoded:

    • "correctness of output" → match the spec literally; raise unless as you go.
    • "code quality" → naming, small methods, single responsibility, idiomatic Ruby.
    • "cleanliness" → no dead code, no commented-out blocks, no over-abstraction.
    • "do not care about efficiency" → don't optimize. O(n) scan over an array is fine.
    • "run your code and iterate" → multi-part. Don't gold-plate Part 1.

    First 90 seconds (do this every time)

    1. Echo the spec back. "So I'm building X that supports Y, and the input is Z — does that match?"
    2. Ask 1–2 clarifying questions max. Common: input format (string commands vs. method calls), error behavior (raise vs. return nil), case sensitivity.
    3. State the data model in one sentence. "I'll model this as a class with a hash of X keyed by Y."
    4. Type the empty skeleton. Class name, one method stub, one inline raise unless. Run it. Now implement.

    Iteration playbook (when interviewer adds a part)

    1. Acknowledge the new requirement aloud before touching code: "OK — now we need to support partial payments. The behavior is X."
    2. Ask: does this change existing behavior, or only add to it? Determines if you refactor or extend.
    3. If refactor needed: say it. "To support this cleanly I'd pull this if chain into a strategy. 2 minutes." Then do it.
    4. Re-run all prior asserts after each part to prove you didn't regress.

    Communication phrases (memorize 3–4 of these)

    OPENING

    "Let me echo back what I heard so we're aligned…"

    "Before I code — quick clarifier: when X happens, do you want Y or Z?"

    WHILE CODING

    "I'm starting with the simplest thing that could possibly work — we can iterate."

    "I'm going to skip input validation for now and come back if we have time."

    WHEN PART 2 LANDS

    "This is going to force me to restructure apply_payment. Bear with me — about 2 minutes."

    "I'll do the simple version first, then we can talk about whether to generalize."

    WHEN STUCK

    "Let me walk through this one input by hand to find where my mental model diverges."

    "I want to verify my assumption — can you confirm that X behaves like Y?"

    Code quality rubric (mental checklist)

    DimensionWhat "good" looks like
    NamingVerbose > clever. apply_payment not apply. remaining_balance not r. Snake_case methods, CamelCase classes.
    Value objectsUse Struct.new(..., keyword_init: true) (or Data.define on Ruby 3.2+) for domain records. Don't pass loose hashes around.
    Method sizeIf a method won't fit on one screen, split it. One reason to exist per method.
    MutationDon't mutate inputs without saying so. Prefer methods that return new objects, or name mutating methods with a ! suffix where idiomatic.
    ErrorsRaise specific exceptions with messages. raise ArgumentError, "no such loan: #{loan_id}".
    TestsInline raise unless after each part. Print "part N ok" on success.
    CommentsAlmost none. One short line for non-obvious WHY. No "this method does X" comments.

    Smells to avoid

    • Nested if/elsif/else > 2 deep — flatten with early returns or a hash dispatch table.
    • Magic numbers/strings — name them as CONSTANTs, or use symbols.
    • Passing raw hashes/arrays around when a Struct would name the fields.
    • Copy-pasted block — extract a method even if it's used twice.
    • Global mutable state — wrap it in a class.
    • One giant method — break by responsibility (parse → validate → apply → format).
    • Float for money — use BigDecimal always.
    • Stale puts / dead code — delete before "I'm done."

    When stuck (recovery moves)

    • Trace one input by hand. Talk it through; the bug usually surfaces.
    • Add a puts right before the failure. See the actual state, not the imagined one.
    • Re-read the spec aloud. 70% of bugs are misread requirements.
    • Ask. "I'm hitting an edge case — when input is X, do you want Y or Z?" Better than thrashing.
    • Don't silently rewrite. Say "I'm going to scrap this and redo with approach B" before you do it.

    Valon-specific framing

    Domain: mortgage servicing. They process payments, track loan state, allocate dollars across buckets (interest, principal, escrow, late fees), and handle different loan types (standard, HELOC, ARM, etc.) with different rules.

    Architectural pattern they're proud of (per their engineering blog): small, reusable allocation rules assembled in different configurations per loan type — i.e. strategy pattern with composable rule chains. If your problem is payment-shaped, lean into this vocabulary.

    Words that will land well: waterfall, allocation, suspense account, idempotency, audit trail, partial payment, principal vs. interest, escrow.

    In-Memory SQL DB most-reported Valon problem

    Problem shape

    Build an in-memory database that supports a few SQL-ish operations on tables. Usually 3–4 parts:

    • Part 1: CREATE TABLE, INSERT, SELECT *.
    • Part 2: WHERE filter (equality, then ranges).
    • Part 3: UPDATE and DELETE.
    • Part 4: Transactions (BEGIN/COMMIT/ROLLBACK) — or indexes, or projection (SELECT col).

    Data model decision

    Keep it simple: rows are hashes keyed by symbol column name, table holds an array of hashes. Don't build a B-tree. Don't parse SQL unless they ask for string input.

    Part 1 — CREATE / INSERT / SELECT *

    class Table
  attr_reader :name, :schema, :rows

  def initialize(name, schema)
    @name   = name
    @schema = schema       # array of column-name symbols, in order
    @rows   = []
  end

  def insert(values)
    if values.length != @schema.length
      raise ArgumentError,
            "#{@name}: expected #{@schema.length} values, got #{values.length}"
    end
    @rows << @schema.zip(values).to_h
  end

  def select_all
    # Return copies so callers can't mutate our rows by accident.
    @rows.map(&:dup)
  end
end


class Database
  def initialize
    @tables = {}
  end

  def create_table(name, schema)
    raise ArgumentError, "table already exists: #{name}" if @tables.key?(name)
    @tables[name] = Table.new(name, schema)
  end

  def table(name)
    @tables[name] or raise KeyError, "no such table: #{name}"
  end
end

    Part 2 — WHERE filter

    Add a where hash and a projection list. Resist the urge to build a query parser.

    class Table
  # SELECT with two optional knobs:
  #   columns: which columns to return  (nil = all columns from the schema)
  #   where:   filter hash, e.g. { age: 30 }  (nil or {} = return every row)
  def select(columns: nil, where: nil)
    cols = columns || @schema           # default projection = every column

    # Walk the rows, keep the ones that match, project down to requested columns.
    results = []
    @rows.each do |row|
      next unless matches?(row, where)  # skip rows that don't match the filter
      results << row.slice(*cols)       # Hash#slice returns a copy with only these keys
    end
    results
  end

  private

  # A row matches when every key/value in `where` is also in the row.
  # No `where` (nil or empty) = no filter, so everything matches.
  def matches?(row, where)
    return true if where.nil? || where.empty?
    where.all? { |column, value| row[column] == value }
  end
end

    Part 3 — UPDATE / DELETE

    Reuse matches?. Return the count of affected rows — that's what SQL does and it's testable.

    class Table
  # UPDATE rows that match a WHERE filter.
  #
  #   set_values: hash of column => new value, e.g. { age: 31 }
  #   where:      same filter shape as SELECT (nil = update every row)
  #
  # Returns the number of rows actually modified (SQL convention — handy in tests).
  def update(set_values, where: nil)
    affected = 0
    @rows.each do |row|
      # Reuse the same matches? helper as SELECT — single source of truth
      # for filter logic. Whatever WHERE means for SELECT, it means here too.
      next unless matches?(row, where)

      # merge! mutates `row` in place: existing keys (like :age) get overwritten,
      # new keys would get added. Because `row` is a reference into @rows, the
      # row stored in the table actually changes — no need to write it back.
      row.merge!(set_values)

      affected += 1
    end
    affected
  end

  # DELETE rows that match a WHERE filter.
  #
  #   where: nil = delete every row (yes, really — same as SQL `DELETE FROM users`)
  #
  # Returns the number of rows actually removed.
  def delete(where: nil)
    # Remember the size before, so we can compute how many we deleted.
    before = @rows.length

    # reject! filters in place: it KEEPS elements where the block returns false,
    # and REMOVES elements where it returns true. So we remove rows that DO match.
    # (Compare to .reject which returns a new array — we'd lose the mutation.)
    @rows.reject! { |row| matches?(row, where) }

    # If nothing matched, reject! returns nil but doesn't change @rows,
    # so `before - @rows.length` is just 0. No special case needed.
    before - @rows.length
  end
end

    Part 4 — Transactions (BEGIN / COMMIT / ROLLBACK)

    Snapshot the whole state on BEGIN. ROLLBACK restores. COMMIT discards the snapshot. Nested transactions = a stack of snapshots. Note: begin is a Ruby keyword, so name the method begin_tx.

    class Database
  def initialize
    @tables = {}

    # Stack of snapshots — one entry per currently-open transaction.
    #   empty stack  = not inside any transaction
    #   N entries    = N nested transactions are open
    # Each snapshot is a full deep copy of @tables at the moment BEGIN was called.
    @snapshots = []
  end

  # Start a new transaction.
  # `begin` is a Ruby keyword, so we name this `begin_tx` to avoid the clash.
  # We save a deep copy of the current state — that's the point we'll roll back to.
  def begin_tx
    @snapshots << deep_dup(@tables)
  end

  # COMMIT just discards the snapshot.
  # Why? The mutations made since BEGIN already happened directly in @tables,
  # so "keeping" them means doing nothing — we just drop the now-unneeded
  # snapshot off the stack.
  def commit
    raise "no transaction in progress" if @snapshots.empty?
    @snapshots.pop
  end

  # ROLLBACK throws away the current state and restores the snapshot.
  # Pop the snapshot we took at BEGIN and make it the live state again.
  # Anything we did since BEGIN is now gone.
  def rollback
    raise "no transaction in progress" if @snapshots.empty?
    @tables = @snapshots.pop
  end

  # (keep create_table / table from Part 1)

  private

  # Marshal.dump serializes any Ruby object to a binary string;
  # Marshal.load reads it back into a brand-new object graph.
  # The round-trip gives us a totally independent deep copy — no references
  # are shared with the original, so mutating one can't affect the other.
  # Slow but trivially correct. (A real DB uses a write-ahead log instead;
  # the spec doesn't care about performance.)
  def deep_dup(obj)
    Marshal.load(Marshal.dump(obj))
  end
end

    Variant — string-command REPL

    If the interviewer hands you commands as strings ("INSERT users 1 Alice 30"), add a thin parser layer that splits on whitespace and dispatches to the methods above. Keep parsing separate from execution.

    def execute(db, command)
  parts = command.split
  op = parts[0].upcase

  case op
  when "CREATE"               # CREATE users id name age
    name, *schema = parts[1..]
    db.create_table(name.to_sym, schema.map(&:to_sym))
  when "INSERT"               # INSERT users 1 Alice 30
    name, *raw = parts[1..]
    db.table(name.to_sym).insert(raw.map { |v| coerce(v) })
  when "SELECT"               # SELECT users
    name = parts[1]
    db.table(name.to_sym).select
  else
    raise ArgumentError, "unknown command: #{op}"
  end
end


def coerce(token)
  return Integer(token) if token.match?(/\A-?\d+\z/)
  return Float(token)   if token.match?(/\A-?\d+\.\d+\z/)
  token
end

    Talking points

    1. Storage choice — array of hashes

    "I'm using an array of hashes for rows — fine at this scale, easy to read. If we needed range queries on a hot column we could add an index, but we're not there yet."

    • Hot column = a column that gets queried, filtered, or joined on far more often than the others. If 90% of WHERE clauses touch user_id, then user_id is hot. "Hot" just means "high traffic" — same metaphor as "hot path" in code.
    • How to add an index if pushed: keep a second hash alongside @rows: @indexes = { user_id: { 42 => [row_ref, row_ref], ... } }. Update it on every insert / update / delete. SELECT with a WHERE on an indexed column does a O(1) hash lookup instead of an O(n) scan.
    • Trade-off to name out loud: indexes speed up reads but slow down writes (extra bookkeeping per mutation) and use more memory. You don't index every column — you index the hot ones.

    2. matches? as single source of truth

    "matches? centralizes the WHERE logic so UPDATE and DELETE reuse SELECT's filter. Single source of truth."

    • Single source of truth = exactly one place in the code defines a rule; everything else refers to that place. If the rule changes, you change it once and every caller picks it up.
    • Concrete payoff: if a later part asks for range filters (e.g. where: { age: 25..30 }), we change matches? alone — SELECT, UPDATE, DELETE all inherit the new semantics. Without centralization that's three copies to keep in sync, and three places a bug can hide.
    • How to extend it for ranges: where.all? { |col, cond| cond.is_a?(Range) ? cond.cover?(row[col]) : row[col] == cond }. Same shape — different per-condition predicate.

    3. Transactions — Marshal deep-clone vs a real WAL

    "For transactions I'm using Marshal.dump/load to deep-clone state — slow but correct. A real DB does this with a journal or WAL; the spec doesn't care about performance."

    • WAL = write-ahead log. Before mutating state, the DB appends a record describing the change to a durable log on disk. If we crash mid-transaction, on restart we replay or undo the log to recover. Used by Postgres, SQLite, MySQL/InnoDB.
    • Journal = the older name for the same idea (filesystem journals work this way too).
    • Why Marshal is slow: it serializes the entire @tables object graph to a binary string and reads it back into new objects. For a 10 MB table that's 10 MB of work on every BEGIN, even if the transaction only touches one row. A WAL writes only the delta — usually tiny.
    • Why Marshal is "trivially correct": works for any data shape, any mutation pattern. We don't have to track which rows were touched or worry about partial writes. Wrong tool at scale, right tool for an interview.
    • If they push for something faster: copy-on-write (only deep-clone a row the first time the tx mutates it) or undo logs (record the inverse of each operation; on rollback, run the inverses in reverse order).

    4. Separating parsing from execution

    "I'm not parsing SQL syntax — the interface is a Ruby method API. If you want a string REPL I can add a thin parser, but I'd keep parsing separate from execution."

    • Why separation matters: the executor can be unit-tested without writing any strings; the parser can be tested without standing up a database. They fail in different ways, so they should be isolated.
    • Swappability: one executor, multiple front-ends. Strings now, JSON over HTTP later, a Ruby DSL on top — all valid input forms that funnel into the same internal call.
    • Validation locality: parse errors ("you wrote an unknown keyword") are a different class of bug from execution errors ("no such table"). Catching parse errors up front means execution code can assume well-formed input.
    • How: the parser produces a plain Ruby object — a hash like { op: :insert, table: :users, values: [1, "Alice"] }, or a Command struct. The executor pattern-matches on op and dispatches to the right method. This is the command pattern in a hat.

    5. Symbols vs strings for column names

    "Symbols for column names because they're identifiers; strings for user-supplied values."

    • Interning = the runtime keeps exactly one copy of a value in memory and reuses it. Ruby interns all symbols: :name written in two different files is the same object in memory (== and equal? both return true).
    • Strings are not interned by default. Every "name" literal allocates a fresh object; two of them are equal by content but distinct objects. (Ruby 3+ can freeze string literals, which helps memory but not equality semantics.)
    • Speed difference: symbol equality is pointer comparison — one CPU op. String equality compares bytes — proportional to length. For a hash with thousands of lookups per second, that adds up.
    • Rule of thumb: symbols for things that look like code you wrote (method names, hash keys you control, column names, enum-like states). Strings for data from outside (user input, file contents, network responses).
    • Why it matters here: row[:name] is a hash lookup against an interned key — fast and identity-checked. If columns were strings, we'd be allocating throwaway lookup keys and comparing bytes on every access.

    Payment Allocation Valon-domain flavored

    Problem shape

    A loan has outstanding balances across several buckets. A payment comes in for some dollar amount. Allocate that amount across the buckets following a waterfall (a priority order). Later parts add: partial payments, different loan types with different waterfalls, fees, escrow.

    Domain model

    Always: use BigDecimal for money (require "bigdecimal"). Never Float. Float rounding in finance code is exactly the kind of thing they're watching for.

    Part 1 — Apply a payment with a simple waterfall

    require "bigdecimal"


# A record of "this much of the payment went into this bucket."
# Struct + keyword_init makes the call site self-documenting:
#   Allocation.new(bucket: :interest, amount: BigDecimal("100"))
# vs the positional form, which forces the reader to remember field order.
Allocation = Struct.new(:bucket, :amount, keyword_init: true)


class Loan
  # The waterfall is the PRIORITY ORDER in which a payment is consumed.
  # Standard rule for mortgages: pay interest before principal.
  # Pulling it out as a constant means the policy is visible at the top
  # of the class, and we can override it per loan type later.
  # .freeze on the array prevents accidental mutation — without it, any
  # caller could do `DEFAULT_WATERFALL.push(:something)` and break us
  # globally.
  DEFAULT_WATERFALL = [:interest, :principal].freeze

  # id is set once at construction → expose only the reader.
  # balances changes with every payment → expose a writer too.
  # waterfall might be swapped (e.g. for testing) → also writable.
  attr_reader   :id
  attr_accessor :balances, :waterfall

  def initialize(id:, balances:, waterfall: DEFAULT_WATERFALL)
    @id = id
    # balances is a hash from bucket symbol → BigDecimal owed in that bucket.
    # e.g. { interest: BigDecimal("100"), principal: BigDecimal("900") }
    @balances  = balances
    @waterfall = waterfall
  end

  # Consume `amount` against the loan's balances, in waterfall order.
  # Returns an array of Allocation records describing where the money went.
  # Mutates @balances in place — afterwards, each bucket reflects what's still owed.
  def apply_payment(amount)
    remaining   = amount   # how much of the payment is still unspent
    allocations = []       # the audit trail we'll return

    # Walk buckets in waterfall order. Each bucket takes as much as it needs,
    # capped by what's left of the payment.
    @waterfall.each do |bucket|
      # .fetch with a default gracefully handles buckets that aren't on this
      # particular loan (e.g. a HELOC has no :escrow bucket). Returns
      # BigDecimal("0") instead of nil, so the comparisons below don't blow up.
      owed = @balances.fetch(bucket, BigDecimal("0"))

      # Skip buckets that are already paid off; stop early when the payment
      # is exhausted. `next` jumps to the next iteration with no else branch.
      next if owed <= 0 || remaining <= 0

      # PARTIAL-ALLOCATION TRICK: take the smaller of "what's owed" and
      # "what's left to spend." If the payment can fully cover this bucket,
      # `owed` wins; otherwise `remaining` wins and the bucket gets a partial.
      applied = [owed, remaining].min

      # Subtract from the bucket (what's still owed) and the payment (what's
      # left to spend). Both are BigDecimal arithmetic — exact, no rounding.
      @balances[bucket] = owed - applied
      remaining -= applied

      allocations << Allocation.new(bucket: bucket, amount: applied)
    end

    allocations
  end
end


# ----- Test: pay $250 against $100 interest + $900 principal -----
# Expected: $100 fully clears interest, the remaining $150 cuts into principal.
loan = Loan.new(
  id: "L1",
  balances: { interest: BigDecimal("100"), principal: BigDecimal("900") }
)
allocs = loan.apply_payment(BigDecimal("250"))

# Two allocations in waterfall order — interest first, principal second.
raise unless allocs == [
  Allocation.new(bucket: :interest,  amount: BigDecimal("100")),
  Allocation.new(bucket: :principal, amount: BigDecimal("150"))
]
# Balances reflect the payment: interest cleared, principal reduced by $150.
raise unless loan.balances[:interest]  == BigDecimal("0")
raise unless loan.balances[:principal] == BigDecimal("750")
puts "part 1 ok"

    Talking points for Part 1

    • BigDecimal, never Float, for money. Floats round silently — 0.1 + 0.2 isn't 0.3 in IEEE 754. BigDecimal stores exact decimal values, which is what a bank statement looks like.
    • apply_payment is an instance method on Loan — the loan knows how to update itself. loan.apply_payment(amount) reads better than Allocator.apply(loan, amount). Tell, Don't Ask.
    • The waterfall is data, not branching code. If we wrote four if/elsif branches instead, adding a fifth bucket means editing the method body. As a list, adding a bucket is a one-line constant change — and Part 2 demonstrates that directly.
    • The [owed, remaining].min trick unifies the "full pay" and "partial pay" cases into one expression. The two cases used to be two branches; the min collapses them.

    Part 2 — Add more buckets (late fees, escrow)

    Only the default waterfall changes. The apply_payment method stays the same — that's the win from modeling it as a list of buckets.

    class Loan
  # Standard mortgage waterfall:
  #   late_fee   — borrower can't dodge late fees by sending a partial
  #   interest   — accrued interest cleared before principal (the lender's rule)
  #   principal  — pay down the actual loan balance
  #   escrow     — money set aside for taxes/insurance; lowest priority
  #                because a missed escrow is less urgent than missed interest
  DEFAULT_WATERFALL = [:late_fee, :interest, :principal, :escrow].freeze

  # (initialize and apply_payment unchanged from Part 1 —
  #  this constant swap is the ENTIRE diff for Part 2)
end

  }
)
# Total owed = $1,000. Payment = $1,000. Should clear everything exactly.
allocs = loan.apply_payment(BigDecimal("1000"))

# Total allocated equals the full payment — nothing left over.
# `sum(BigDecimal("0"), &:amount)` is shorthand for `allocs.sum { |a| a.amount }`
# with a BigDecimal seed. The seed matters: without it, Array#sum starts at
# Integer 0, and mixing Integer + BigDecimal in the reduce gets weird.
# `&:amount` is the symbol-to-proc shortcut for `{ |a| a.amount }`.
raise unless allocs.sum(BigDecimal("0"), &:amount) == BigDecimal("1000")

# Every bucket fully paid off — all balances are zero.
raise unless loan.balances.values.all? { |v| v == 0 }
puts "part 2 ok"

    Talking points for Part 2

    • "Only the constant changed; the method didn't." Say this out loud right after you make the edit. It's the dividend of modeling the waterfall as data, and naming it earns interview points.
    • The order of buckets IS the business policy. Reordering changes the economics for both borrower and lender. Late-fee-first is a borrower-unfriendly rule that prevents the borrower from "starving" the late fee bucket; interest-before-principal is the standard mortgage convention. If you're asked "why this order?" — that's the answer.
    • Symbols for bucket names for the same reason we used symbols for column names in the SQL DB tab — they're identifiers we control, not user data, and Ruby interns them so comparison is pointer equality.
    • The Array#sum seed argument is a subtle Ruby gotcha worth knowing: [BigDecimal("1")].sum is fine because Array#sum reduces with +, but the initial accumulator is Integer 0 by default. Passing BigDecimal("0") as the seed keeps the accumulator type consistent.

    Part 3 — Partial payments → suspense account

    If the payment is less than the full required amount, don't allocate anything — hold it in a suspense account until enough has accumulated to cover the required payment. This is real mortgage servicing behavior.

    # apply_payment used to return just an array of Allocations.
# Now it needs to communicate three possible outcomes: what was allocated,
# what got held in suspense, and any overpayment. A Struct with named fields
# is self-documenting at the call site — much better than a 3-tuple where
# the reader has to remember position.
PaymentResult = Struct.new(:allocations, :held_in_suspense, :overpayment, keyword_init: true)


class Loan
  DEFAULT_WATERFALL = [:late_fee, :interest, :principal, :escrow].freeze

  attr_reader   :id
  attr_accessor :balances, :waterfall, :suspense

  def initialize(id:, balances:, waterfall: DEFAULT_WATERFALL)
    @id        = id
    @balances  = balances
    @waterfall = waterfall
    # Suspense is money the loan has RECEIVED but not yet APPLIED. It sits
    # in a holding account because the borrower didn't send enough to cover
    # a full required payment. Starts at zero; grows when partials come in;
    # resets to zero the moment a payment finally clears the threshold.
    @suspense = BigDecimal("0")
  end

  # Total dollars currently owed across every bucket.
  # Used to decide whether an incoming payment is "enough" to apply or
  # whether we should hold it in suspense.
  def total_owed
    @balances.values.sum(BigDecimal("0"))
  end

  def apply_payment(amount)
    # Combine any previously-held suspense with the new payment.
    # Example: borrower previously sent $200 (held in suspense) and now sends
    # $300 — together we have $500 to consider against `total_owed`.
    funds = @suspense + amount

    # ---- PARTIAL-PAYMENT RULE ----
    # If combined funds STILL don't cover the full required payment, don't
    # allocate ANYTHING. Park it in suspense until enough has accumulated.
    #
    # Why hold rather than apply a partial?
    # Because applying a partial would make the loan look partially current
    # when it isn't. Downstream systems that ask "is this loan delinquent?"
    # would get a misleading answer. Holding preserves the honest state.
    if funds < total_owed
      @suspense = funds
      return PaymentResult.new(
        allocations:      [],
        held_in_suspense: funds,
        overpayment:      BigDecimal("0")
      )
    end

    # ---- ENOUGH FUNDS — RELEASE SUSPENSE AND APPLY ----
    # Reset suspense to zero now that we're about to flush it through the
    # waterfall. After this point the code is the same shape as Part 1/2,
    # except `remaining` starts at `funds` (suspense + payment) instead of
    # just `amount`.
    @suspense = BigDecimal("0")

    allocations = []
    remaining   = funds

    @waterfall.each do |bucket|
      owed = @balances.fetch(bucket, BigDecimal("0"))
      next if owed <= 0 || remaining <= 0
      applied = [owed, remaining].min
      @balances[bucket] = owed - applied
      remaining -= applied
      allocations << Allocation.new(bucket: bucket, amount: applied)
    end

    # Whatever's left in `remaining` after every bucket is paid is an
    # OVERPAYMENT — the borrower sent more than was owed. Real systems put
    # this somewhere (a credit on the next bill, a refund, or applied to
    # additional principal). We just surface it in the result and let the
    # caller decide.
    PaymentResult.new(
      allocations:      allocations,
      held_in_suspense: BigDecimal("0"),
      overpayment:      remaining
    )
  end
end


# ----- Test: two partial payments combine to clear the loan -----
loan = Loan.new(
  id: "L3",
  balances: { interest: BigDecimal("100"), principal: BigDecimal("400") }
)
# Total owed = $500.

# First payment: $200. Not enough to cover $500 — held in suspense.
# Nothing gets allocated; the balances are unchanged.

raise unless r1.allocations.empty?
raise unless r1.held_in_suspense == BigDecimal("200")

# Now we hit the threshold — suspense releases and $500 flows through
# the waterfall.
r2 = loan.apply_payment(BigDecimal("300"))
raise unless r2.allocations.sum(BigDecimal("0"), &:amount) == BigDecimal("500")
raise unless loan.suspense == BigDecimal("0")
raise unless loan.balances.values.all? { |v| v == 0 }
puts "part 3 ok"

    Talking points for Part 3

    • Suspense is a real mortgage-servicing concept, not just a design choice. Valon's own customer help pages describe it: a partial payment "isn't enough to satisfy the monthly amount due, so it's held in a suspense account until enough has accumulated." Saying "this matches how Valon explains payments on their help center" is a strong move that signals you did domain homework.
    • Why hold instead of apply: if we applied a $200 partial across buckets, the loan would look 40% paid even though the borrower still owes a full month. Downstream "is this loan delinquent?" systems would give a misleading answer. Holding keeps the delinquency state honest, which is what compliance and collections workflows depend on.
    • Why PaymentResult instead of a tuple: apply_payment now has three possible outcomes to communicate (allocations, suspense, overpayment). Named fields read better at the call site (result.held_in_suspense) and don't break callers if a fourth field is added later. With a positional tuple, every caller has to flip back to the function definition.
    • Overpayment vs. prepayment: mention you'd typically route overpayment to additional principal (an "extra principal payment"), but you're surfacing it in the result instead so the caller can decide. Shows you've thought about the next layer up.
    • Why total_owed is its own method: it's used twice (in the if-check and conceptually in the test), and naming the concept ("total currently owed") makes the code readable. If asked "what if total_owed got expensive?" — you'd memoize it, but at this scale just computing it each time is fine.

    Part 4 — Different loan types (HELOC etc.)

    HELOC follows a different waterfall (interest → late fees → principal), and applies partials immediately (no suspense hold). Different loan type = different Rules object. The core apply_payment stays the same.

    # A LoanRules object captures the policy bits that VARY between loan types.
# It's a Struct because rules are DATA, not behavior: we don't need
# subclasses with overridden methods, we just need a record that holds
# the configs.
#
# The `do ... end` block at the end of Struct.new lets us add methods to
# the resulting class. alias_method gives us a `.hold_partials?` predicate
# — Ruby's convention is that boolean methods end in `?`.
# Without the alias, callers would write `rules.hold_partials` which doesn't
# read as a boolean question.
LoanRules = Struct.new(:waterfall, :hold_partials, keyword_init: true) do
  alias_method :hold_partials?, :hold_partials
end

# Standard mortgage: full four-bucket waterfall, holds partials in suspense.
STANDARD_RULES = LoanRules.new(
  waterfall:     [:late_fee, :interest, :principal, :escrow].freeze,
  hold_partials: true
)

# HELOC (Home Equity Line Of Credit) servicing rules differ:
#   - The waterfall is shorter (no separate escrow bucket).
#   - Interest goes first, even before late fees.
#   - Partials are applied IMMEDIATELY — the lender wants to collect
#     whatever the borrower sends right away, not park it in suspense.
HELOC_RULES = LoanRules.new(
  waterfall:     [:interest, :late_fee, :principal].freeze,
  hold_partials: false
)


class Loan
  # rules is set at construction and shouldn't change — reader only.
  attr_reader   :id, :rules
  attr_accessor :balances, :suspense

  def initialize(id:, rules:, balances:)
    @id       = id
    @rules    = rules            # the LoanRules object that controls behavior
    @balances = balances
    @suspense = BigDecimal("0")
  end

  def total_owed
    @balances.values.sum(BigDecimal("0"))
  end

  # Same shape as Part 3, but the policy bits read from @rules instead of
  # being hard-coded. The allocation engine itself is UNCHANGED.
  def apply_payment(amount)
    funds = @suspense + amount

    # The ONLY behavior change between loan types: do we hold partials?
    # A standard mortgage says yes; a HELOC says no.
    if @rules.hold_partials? && funds < total_owed
      @suspense = funds
      return PaymentResult.new(
        allocations:      [],
        held_in_suspense: funds,
        overpayment:      BigDecimal("0")
      )
    end

    @suspense = BigDecimal("0")
    allocations = []
    remaining   = funds

    # The OTHER policy bit: which buckets, in what order. Read from rules.
    # The loop body itself is identical to every prior part.
    @rules.waterfall.each do |bucket|
      owed = @balances.fetch(bucket, BigDecimal("0"))
      next if owed <= 0 || remaining <= 0
      applied = [owed, remaining].min
      @balances[bucket] = owed - applied
      remaining -= applied
      allocations << Allocation.new(bucket: bucket, amount: applied)
    end

    PaymentResult.new(
      allocations:      allocations,
      held_in_suspense: BigDecimal("0"),
      overpayment:      remaining
    )
  end
end


# ----- Test: HELOC accepts a partial and applies it immediately -----
heloc = Loan.new(
  id: "H1",
  rules: HELOC_RULES,
  balances: { interest: BigDecimal("50"), principal: BigDecimal("450") }
)
# Total owed = $500. Send only $100 (clearly a partial).
# Because HELOC_RULES.hold_partials? is false, we DON'T suspense it —
# we run it through the waterfall right away.

raise unless r.allocations.sum(BigDecimal("0"), &:amount) == BigDecimal("100")
# Interest ($50) was first in the HELOC waterfall and got fully cleared.

raise unless heloc.balances[:interest]  == BigDecimal("0")
raise unless heloc.balances[:principal] == BigDecimal("400")

    Talking points for Part 4

    • "The engine didn't change." All four parts share the exact same waterfall loop. New loan types are new Rules objects, not new branches inside apply_payment. This is the modular-rules approach Valon's engineering blog calls out — name-drop it: "this is the same composable-rules pattern your blog describes."
    • Why duck-typed Rules instead of a class hierarchy: any object that responds to .waterfall and .hold_partials? works. Inheritance would have locked us into a particular shape and made testing harder (you'd need test doubles for the abstract base class). With duck typing, you can pass a plain Struct, a custom class, or even a stub in tests — whatever has the right shape.
    • The ? convention in Ruby: boolean methods conventionally end in ? (empty?, nil?, include?). The alias_method :hold_partials?, :hold_partials trick gets us that without writing a second method body. The do block on Struct.new is the place to add methods to the resulting class.
    • What you'd add next: ARM_RULES for adjustable-rate (waterfall might depend on the current rate period), REVERSE_MORTGAGE_RULES (no monthly payment at all), FHA_RULES (extra mortgage-insurance bucket). Each is one new constant. The engine doesn't care.
    • If they push on extensibility: the limit of this design is when rules need to express things that aren't just "a list of buckets and a boolean." E.g. "for FHA loans, route MIP partials to escrow even if late fees are owed." That's a per-bucket conditional, not a per-loan one — you'd grow LoanRules to hold a small set of named predicates, or extract bucket-level allocation strategies. Worth flagging as the next refactor without doing it preemptively.

    Talking points

    • "I'm using BigDecimal everywhere — never Float for money. Float rounding errors compound silently."
    • "The waterfall is data, not code — adding a new loan type is a new Rules object, not a new branch in apply_payment. This matches Valon's modular-loan-rules approach." (Says you've read their engineering blog without making it weird.)
    • "Suspense is a real-world mortgage concept — partial payments aren't applied because the loan would otherwise look 'cured' when it isn't. Holding in suspense preserves the delinquency state."
    • "PaymentResult is a Struct with named fields rather than a raw hash, because callers shouldn't have to remember keys or risk typos."
    • "apply_payment is an instance method on Loan — Tell, Don't Ask. The loan knows how to apply a payment to itself."
    • "I'm not handling currency conversion, posting dates, or backdating — those are real concerns but out of scope for this hour."

    OO / Pluggable Components Rippling-style; common shape across companies

    Problem shape

    Build a system with a core orchestrator (Logger) and pluggable workers (Handlers). Later parts add filter/transform layers and combinations. The point is composition over inheritance — small handlers that wrap other handlers, like middleware.

    Part 1 — Basic logger + console handler

    LogEvent = Struct.new(:level, :message, keyword_init: true)


# Base "interface" — duck-typed; subclasses must implement #handle.
class Handler
  def handle(event)
    raise NotImplementedError, "#{self.class} must implement #handle"
  end
end


class ConsoleHandler < Handler
  attr_reader :received

  def initialize
    @received = []           # captured for tests
  end

  def handle(event)
    @received << event
    puts "[#{event.level}] #{event.message}"
  end
end


class Logger
  def initialize
    @handlers = []
  end

  def add(handler)
    @handlers << handler
    self                      # chainable
  end

  def log(level, message)
    event = LogEvent.new(level: level, message: message)
    @handlers.each { |h| h.handle(event) }
  end
end


# Test
console = ConsoleHandler.new

raise unless console.received == [LogEvent.new(level: "INFO", message: "hello")]
puts "part 1 ok"

  # Useful for tests and for buffering before flushing somewhere else.
  attr_reader :events

  def initialize
    @events = []
  end

  def handle(event)
    @events << event
  end
end


class FileHandler < Handler
  def initialize(path)

  def handle(event)
    File.open(@path, "a") { |f| f.puts "[#{event.level}] #{event.message}" }

end

    Part 3 — Filter / transform layers (decorator pattern)

    The key move: a handler that wraps another handler. This is composition, not inheritance — you can stack filters and transforms in any order.

    LEVEL_RANK = { "DEBUG" => 0, "INFO" => 1, "WARN" => 2, "ERROR" => 3 }.freeze


class LevelFilter < Handler
  # Forwards events at or above min_level to the wrapped handler.
  def initialize(inner, min_level:)
    @inner = inner
    @min   = LEVEL_RANK.fetch(min_level)
  end

  def handle(event)
    @inner.handle(event) if LEVEL_RANK.fetch(event.level) >= @min
  end
end


class Redactor < Handler
  # Transform: replace each pattern in the message with [REDACTED].
  def initialize(inner, patterns:)
    @inner    = inner
    @patterns = patterns
  end

  def handle(event)
    msg = event.message
    @patterns.each { |p| msg = msg.gsub(p, "[REDACTED]") }
    @inner.handle(LogEvent.new(level: event.level, message: msg))
  end
end


# Test — stack the layers. INFO+ goes to memory, with SSN redacted.
mem = MemoryHandler.new
pipeline = LevelFilter.new(
  Redactor.new(mem, patterns: ["123-45-6789"]),
  min_level: "INFO"
)
log = Logger.new.add(pipeline)

log.log("DEBUG", "ignored")                          # filtered out
log.log("INFO",  "user ssn=123-45-6789 logged in")   # redacted, kept
raise unless mem.events.length == 1
raise unless mem.events.first.message == "user ssn=[REDACTED] logged in"
puts "part 3 ok"

    Part 4 — Batched / fan-out handlers

    If they ask for batching: a handler that buffers and flushes every N events. If they ask for fan-out: a handler that holds a list of inner handlers.

    class BatchHandler < Handler
  # Buffer events; flush to inner in batches of size N.
  def initialize(inner, batch_size:)
    @inner      = inner
    @batch_size = batch_size
    @buffer     = []
  end

  def handle(event)
    @buffer << event
    flush if @buffer.length >= @batch_size
  end

  def flush
    @buffer.each { |event| @inner.handle(event) }
    @buffer.clear
  end
end


class FanOut < Handler
  # Send every event to all inner handlers.
  def initialize(*inners)
    @inners = inners
  end

  def handle(event)
    @inners.each { |inner| inner.handle(event) }
  end
end

    Talking points

    • "Handler is the only abstraction — filters, transforms, batchers, and fan-outs are all just handlers that wrap another handler. That's composition over inheritance."
    • "The order matters: LevelFilter.new(Redactor.new(mem)) redacts before filtering; Redactor.new(LevelFilter.new(mem)) filters before redacting. I'd make this explicit in real code with a builder."
    • "LogEvent is a Struct; if I wanted enforced immutability I'd use Data.define on Ruby 3.2+."
    • "Ruby is duck-typed so I don't strictly need a Handler base class — anything that responds to #handle works. I'm including one to make the contract explicit and to fail loudly if a subclass forgets to implement it."
    • "If we needed async fan-out I'd put a queue in front of the workers — but the spec is sync so I'm not building that."

    String-command parser (when input is text)

    HANDLERS = {
  "ADD" => ->(state, k, v) { state.add(k, coerce(v)) },
  "GET" => ->(state, k)    { state.get(k) },
  "DEL" => ->(state, k)    { state.delete(k) }
}.freeze


def execute(state, command)
  op, *args = command.split
  handler = HANDLERS[op.upcase] or raise ArgumentError, "unknown command: #{op}"
  handler.call(state, *args)
end


def coerce(token)
  return Integer(token) if token.match?(/\A-?\d+\z/)
  return Float(token)   if token.match?(/\A-?\d+\.\d+\z/)
  token
end
