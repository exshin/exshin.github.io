    How to answer a short values round

    • Pick fast, keep it ~2 min. "Let me think of the best example" (5s) beats a 5-minute ramble. He may only get through 3–4 questions — make each count.
    • STAR, but land the Result. Situation (1 line) → your role → the hard decision → outcome + what you learned.
    • Name the humans. "I pulled in the PM and support early," "two VPs in different orgs," "the architecture group." Collaboration is invisible unless you name who you collaborated with.
    • One value, one or two stories. The same story can answer several prompts — bend the framing to the question instead of inventing new material.
    • Be honestly self-aware on failure. Own your part, no blaming, state the concrete change since. That is the answer.

    Drive a 60-min design round (the spine)

    Spend the time roughly like this. Narrate which phase you're in.

    Phase~minWhat you do
    1. Requirements8Functional + non-functional. Pin scale numbers. State what's out of scope.
    2. API + data model10A few endpoints, then the tables/entities. The schema is where payments rounds are won.
    3. High-level architecture10Boxes & arrows: clients → API → services → stores → queues. Happy path first.
    4. Deep dive (1–2)20Where the meat is — idempotency, concurrency on balances, consistency, the write path. Let them steer.
    5. Failure & scale8What breaks: retries, partial failure, hot accounts, partitioning, dedup. Bottlenecks & fixes.
    6. Wrap4Recap the 2–3 key tradeoffs you made and what you'd revisit with more time.

    Golden rule: don't draw until you've agreed on requirements. Jumping to a diagram is the #1 thing that reads as junior.

    Clarify before you draw

    Functional: who/what are the actors? What's the core operation (transfer? charge? settle?)? Single-currency or FX? Do funds need to be held/authorized then captured, or one-shot? Refunds/reversals?

    Non-functional: scale (transfers/sec, accounts, read:write ratio)? Latency budget? Consistency requirement (almost always strong for balances)? Availability target? Audit/regulatory needs (immutable history)?

    Payments-specific things to surface (this is the signal):

    • Idempotency: "Clients will retry — do we get an idempotency key, or do I design one?"
    • Double-spend / overdraft: "Can a balance go negative? How do we prevent two concurrent transfers from over-drawing?"
    • Source of truth & auditability: "Is the ledger the system of record? I'll make it append-only and immutable."
    • Money never lost: "Every cent debited lands somewhere — I'll model this double-entry so the books always balance."

    Asking these 4 unprompted is the fastest way to show a Staff interviewer you've built real payment systems.

    Idempotency & exactly-once

    Networks retry. Without dedup, a retried "transfer $100" charges twice. Exactly-once = at-least-once delivery + idempotent processing. You can't guarantee exactly-once delivery, so you make the operation idempotent.

    • Idempotency key: client sends a unique key per intent (e.g. UUID). Server stores (key → result) in a table with a unique constraint. First request does the work + records the result; retries return the stored result without re-doing it.
    • Atomicity: insert the idempotency record in the same transaction as the ledger write, so you can't record "done" without doing it (or vice-versa).
    • Race: two concurrent requests with the same key — the unique constraint makes one win; the loser waits/retries and reads the result.
    • Scope & TTL: key is unique per endpoint/account; keep records long enough to cover client retry windows (often 24h+).

    Say it Ripple-style: "A 200 must mean it actually happened exactly once — not 'we received your request.' I've shipped exactly this; I had a prod bug where a success masked a no-op." (your silent-200 story)

    Double-entry ledger (the heart of it)

    A double-entry ledger is append-only: you never mutate a balance in place — you write immutable entries, and every transaction's debits equal its credits. Balance = sum of entries for an account. This makes the system auditable and makes "lost money" structurally impossible.

    -- Accounts (one row per money bucket: user wallet, fee account, etc.)
accounts(
  id            BIGINT PK,
  currency      CHAR(3),
  -- optional cached balance for fast reads, derived from entries
  balance_minor BIGINT NOT NULL DEFAULT 0
)

-- A transfer = a logical money movement (the "intent")
transfers(
  id              BIGINT PK,
  idempotency_key TEXT UNIQUE NOT NULL,
  status          TEXT,        -- pending | posted | failed
  created_at      TIMESTAMPTZ
)

-- Ledger entries: immutable, append-only. 2+ rows per transfer.
-- Sum of amount_minor for a given transfer_id must equal 0.
ledger_entries(
  id          BIGINT PK,
  transfer_id BIGINT REFERENCES transfers(id),
  account_id  BIGINT REFERENCES accounts(id),
  amount_minor BIGINT NOT NULL,  -- +credit / -debit, integer minor units
  created_at  TIMESTAMPTZ,
  UNIQUE(transfer_id, account_id)
)
    • Money as integers (minor units / cents), never floats — floats lose pennies.
    • Debits negative, credits positive; each transfer's entries sum to zero → the system never creates or destroys value.
    • Balance = cached column updated in the same txn, or derived as SUM(amount_minor) (slower but verifiable). Best: cache it, and periodically reconcile cache vs sum.
    • Immutability: corrections are new reversing entries, never edits/deletes. The history is the audit trail.

    Consistency, transactions & locking

    Balances need strong consistency — eventual consistency lets two transfers both read an old balance and overdraw. Two standard ways to make concurrent debits safe on the same account:

    • Pessimistic (row lock): SELECT ... FOR UPDATE on the account row inside the txn. Simple, correct, but the locked row is a hot point for popular accounts.
    • Optimistic (version / CAS): read balance + version, write with WHERE version = $read; if 0 rows updated, someone beat you → retry. Better under low contention.

    Always wrap the whole money move in ONE DB transaction: insert transfer + insert ledger entries + update balances + insert idempotency record, all-or-nothing. Use a strong isolation level (e.g. serializable or repeatable read) for the balance-critical path.

    CAP framing: "For money, I choose consistency over availability — I'd rather reject a transfer than risk a double-spend. CP, not AP."

    Saga vs 2-phase commit (money across services)

    When a transfer spans services/databases (debit service A, credit service B), you can't use one DB transaction. Two patterns:

    • 2-phase commit (2PC): a coordinator asks all participants to "prepare," then "commit." Strong consistency but blocking — if the coordinator dies mid-flight, participants are stuck holding locks. Rarely used at scale.
    • Saga (preferred): a sequence of local transactions, each with a compensating action. Debit A (hold), credit B; if credit B fails, run the compensation (release/refund the hold on A). Eventually consistent, non-blocking, resilient. Orchestrated (central coordinator) or choreographed (events).
    • Holds / reservations: for cross-service money, debit as a pending hold first, capture on success, release on failure — so funds are never double-committed.

    Ripple tie-in: cross-border settlement is exactly this — you reserve liquidity, settle, and need clean compensation if a leg fails. Sagas + idempotent compensations are the daily reality of money rails.

    Reconciliation

    Reconciliation = continuously proving two records of truth agree. It's how you catch the bug before it reaches a customer's statement.

    • Internal: cached balance == SUM(ledger entries)? Do all transfers' entries sum to zero? Run as a periodic job; alert on any drift.
    • External: our ledger vs the bank/processor/on-chain record. Match by reference id; flag unmatched/mismatched into an exceptions queue for investigation.
    • Design hook: immutable append-only entries + stable reference ids make reconciliation tractable. This is why you don't mutate balances in place.

    Outbox & event-driven

    Problem: you write to the DB and need to publish an event (e.g. "transfer.posted") — but a crash between the two loses the event or double-publishes.

    • Transactional outbox: in the same DB txn as the ledger write, insert a row into an outbox table. A separate relay reads the outbox and publishes to the queue, marking rows sent. Atomic with the write, at-least-once to consumers.
    • Consumers must be idempotent (event carries an id) since delivery is at-least-once.
    • Use for: notifications, webhooks, downstream ledger projections, analytics — anything that reacts to a money event.

    ★ Worked: Idempotent money-transfer API

    Prompt: "Design a service that transfers money between two accounts." This is the anchor — almost everything else is a variation. Walk it in the 6-phase spine.

    1. Requirements. Move funds A→B, same currency. Strongly consistent balances, no overdraft, no double-charge on retry, full audit trail. Scale: say 1–5k transfers/sec, read-heavy on balances. Out of scope (state it): FX, KYC, fraud.

    2. API.

    POST /transfers
  Idempotency-Key: <uuid>
  { from_account, to_account, amount_minor, currency }
  -> 201 { transfer_id, status: "posted" }   // or replay of prior result

GET /transfers/{id}        -> status + entries
GET /accounts/{id}/balance -> current balance

    3. Data model. The ledger schema above: accounts, transfers (with idempotency_key UNIQUE), ledger_entries (two rows: −amount on A, +amount on B).

    4. The write path (the deep dive). One DB transaction:

    1. Insert into transfers with the idempotency key. If unique-violation → it's a retry: return the existing transfer's result. Done.
    2. Lock both account rows in a consistent order (e.g. ascending id) to avoid deadlocks — SELECT ... FOR UPDATE.
    3. Check from balance ≥ amount; else fail (insufficient funds).
    4. Insert two ledger_entries (−amount, +amount); they sum to 0.
    5. Update both cached balances. Mark transfer posted. Commit.

    5. Failure & scale.

    • Retry mid-flight: idempotency key + atomic txn → safe, no double-charge.
    • Deadlocks: always lock accounts in sorted order.
    • Hot account (e.g. a fee/treasury account everyone credits): the row lock serializes everything → bottleneck. Mitigate with balance sharding (N sub-balances summed) or batching.
    • Scaling reads: balance read replicas / cache, but reads that gate a transfer must hit primary.
    • Partitioning: shard by account id; cross-shard transfers become a saga (see above).
    • Emit a transfer.posted event via the outbox for notifications/downstream.

    6. Wrap tradeoffs: "Single-DB transaction for correctness while it fits one node; row locks for simplicity, with sharded balances for hot accounts; saga only when I must cross shards — because each hop away from one transaction is a hop away from easy correctness, and in payments correctness is the product."

    Double-entry ledger service

    If asked for the ledger specifically (vs the transfer API on top of it): emphasize it as the immutable system of record that other services post to.

    • API: POST /entries takes a balanced set of debits/credits (must sum to 0) atomically; reject if unbalanced.
    • Append-only; no updates/deletes; corrections are reversing entries.
    • Balances are a projection (cached column or materialized view) reconciled against the entry sum.
    • Partition entries by account; keep a stable external_ref for reconciliation.
    • Tradeoff to name: derive-balance-on-read (always correct, slow) vs cached-balance (fast, needs reconciliation) — pick cached + reconcile.

    Cross-border payment / settlement (Ripple's actual domain)

    If they go domain-specific: design moving value from a payer in country X to a payee in country Y, different currencies. This is literally Ripple Payments / ODL.

    • The hard parts: two currencies (FX), two ledgers/rails that settle on different timelines, liquidity that must be available on the destination side, and a payment that must either fully complete or fully unwind.
    • Shape: quote (lock an FX rate, with expiry) → reserve source funds + destination liquidity → execute both legs → confirm settlement finality → release. Model as a saga with compensations per leg.
    • Bridge asset / liquidity: classic correspondent banking pre-funds nostro/vostro accounts (trapped capital). Ripple's pitch: use a bridge asset (XRP) or stablecoin (RLUSD) so you source liquidity on demand instead of pre-funding — less trapped capital, seconds not days.
    • Settlement finality: on a blockchain leg, "done" means the ledger confirmed/validated — design around the chain's finality, not an optimistic ack.
    • Idempotency & reconciliation across both legs and external rails are the whole game.

    You don't need XRPL internals — know the shape: quote → reserve → settle legs → confirm finality → compensate on failure, with on-demand liquidity replacing pre-funded accounts.

    Digital wallet / balance service

    • Same ledger core; wallet = an account (or set of per-currency accounts) per user.
    • Operations: deposit (external → wallet), withdraw (wallet → external), internal transfer.
    • Deposits/withdrawals cross an external rail → saga + holds; internal transfers are one DB txn.
    • Available vs pending balance: pending holds reduce available without yet posting — model holds explicitly.
    • Strong consistency on the spend path; eventual is fine for display/history.

    Reconciliation pipeline

    • Ingest both sides (internal ledger export + external statement / on-chain records) into a staging store.
    • Match by reference id + amount + date; bucket into matched / missing-on-one-side / amount-mismatch.
    • Exceptions → a queue/dashboard with assignment + audit; auto-resolve known patterns.
    • Run incrementally (windowed) for scale; idempotent so re-runs don't double-count.
    • Metrics: unmatched count/value, time-to-resolve, drift trend → alert on anomalies.

    Rate limiter (classic, non-payments)

    A common warm-up even at payments shops (protect a money API from abuse/ret* storms).

    • Algorithms: token bucket (allows bursts, most common), leaky bucket (smooths), fixed/sliding window.
    • Distributed: counters in Redis keyed by client + window; atomic INCR/Lua to avoid races; TTL the keys.
    • Tradeoffs: fixed-window has boundary bursts; sliding-window-log is accurate but memory-heavy; sliding-window-counter is the usual compromise.
    • Decisions: per-user vs per-IP vs per-endpoint; fail-open vs fail-closed (for a money API, lean fail-closed on critical paths).

    Webhook / notification delivery

    • Source events from the outbox (atomic with the money write) → queue.
    • Workers deliver with retries + exponential backoff; dead-letter after N tries.
    • At-least-once delivery → sign payloads + include an event id so consumers dedup (idempotent receivers).
    • Ordering: usually per-entity ordering only; use partition keys if order matters.
    • Track delivery status per subscriber; expose a replay endpoint.

    Tradeoffs a Staff interviewer probes

    • Strong vs eventual consistency — and where each is acceptable (spend path strong; history/display eventual).
    • Single DB txn vs distributed saga — the correctness cost of every hop across a boundary.
    • Pessimistic vs optimistic locking under low vs high contention; hot-account mitigation.
    • Cached balance vs derived — speed vs verifiability, reconciled by a job.
    • Sync vs async — what the caller must see commit synchronously vs what can ride the outbox.
    • Sharding key — by account id; what breaks (cross-shard transfers) and how you handle it.

    There's rarely one right answer — they're scoring whether you can name the axis, pick a point, and defend it for this problem.

    Ripple domain hooks to drop

    • "In money movement I optimize for correctness first — exactly-once and a ledger that always balances — then latency, then throughput."
    • "This is the same problem I solve in billing: idempotent retries, source-of-truth conflicts, reconciliation."
    • "For cross-border, on-demand liquidity beats pre-funded nostro/vostro — less trapped capital, faster settlement."
    • "Settlement finality means I design around the ledger confirming, not an optimistic ack."
    • Vocab: idempotency, double-entry, settlement finality, liquidity, reconciliation, saga/compensation, append-only, exactly-once.

    Red flags & tips

    • Drawing before scoping. Always lock requirements + scale first.
    • Floats for money. Integers in minor units, always.
    • Mutating balances in place with no audit trail. Append-only ledger; balance is derived/reconciled.
    • Ignoring retries / concurrency. If you don't bring up idempotency and overdraft races yourself, you've missed the point of a payments round.
    • Hand-waving "I'll use a distributed transaction." Know 2PC's blocking failure mode and why sagas usually win.
    • Going silent. Think out loud — the reasoning is what's scored. "Let me weigh two options here…"
    • Over-engineering for scale you weren't asked about. Start simple (one DB, correct), scale when they push.

    Data-structure design — the key bucket

    If one problem type is worth over-practicing, it's this. Drill the muscle of: pick the right backing structures, keep every operation at target complexity, handle edge cases (empty, eviction, duplicate keys).

    • LRU Cache — hashmap + doubly-linked list, all O(1). Know it cold.
    • O(1) insert / remove / getRandom — hashmap + array swap-to-end.
    • insert / get / getAverage in O(1) — running sum + count (reported prompt).
    • Stock-price tracker — current/max/min with a heap + lazy deletion or a tree map.
    • Rate limiter / token bucket — ties straight into the design tab; a nice bridge answer.

    Communication framework (coding)

    1. Restate the problem in your own words.
    2. Clarify 2–3 things (input size, edge cases, can I mutate, sorted?).
    3. Brute force first — state it & its complexity.
    4. Outline the better approach before coding; get a nod.
    5. Narrate non-obvious decisions.
    6. Trace a concrete example after coding.
    7. State complexity (time + space) unprompted, then edge cases.

    Payments-flavored coding twists

    • Money math: if amounts appear, use integers (minor units), never floats — and say why.
    • Idempotency / dedup as a coding sub-problem: "process each event once" → a seen-set keyed by id.
    • Concurrency follow-up: "what if two threads call this?" → mention a lock / atomic / CAS even if not coded.
    • Interval/scheduling framings can wear a payments coat (overlapping holds, settlement windows).
